## 🎯 Главный вопрос: Почему id() не меняется?

**Краткий ответ:** `id()` возвращает адрес **объекта-списка**, а не адрес **массива элементов**. При переполнении перемещается только внутренний массив, а сам объект остаётся на месте.

## 💡 Почему так сделано?

### Преимущества двухуровневой структуры:

1. **Стабильность ссылок**

python

```python
   my_list = [1, 2, 3]
   reference = my_list  # Ссылка на объект
   
   my_list.append(4)
   my_list.extend([5, 6, 7, 8])  # Много добавлений
   
   # reference всё ещё указывает на тот же объект!
   print(reference is my_list)  # True
   print(id(reference) == id(my_list))  # True
```

2. **Эффективная работа словарей**

python

```python
   data = {}
   my_list = [1, 2, 3]
   
   data[id(my_list)] = "some value"
   
   my_list.append(4)  # Внутренний массив переехал
   
   # Но ключ словаря всё ещё работает!
   print(data[id(my_list)])  # "some value"
```

3. **Консистентность в многопоточности**
    - ID объекта не меняется
    - Проще синхронизировать доступ

## 📈 Стратегия роста списков

Python использует формулу роста (упрощённо):

python

```python
new_allocated = (current_size + (current_size >> 3) + 6)
```

**🛠️ Практически это даёт:**

|Текущий размер|Новая ёмкость|Рост|
|---|---|---|
|0|4|—|
|4|8|2x|
|8|16|2x|
|16|25|1.56x|
|25|35|1.4x|
|35|46|1.31x|
|64|80|1.25x|
|128|161|1.26x|

Со временем коэффициент роста **стремится к ~1.125x** для больших списков.

## 🔄 Когда ID ДЕЙСТВИТЕЛЬНО меняется?

**⚠️ Только при создании НОВОГО объекта:**

python

```python
lst = [1, 2, 3]
print(f"ID: {id(lst)}")  # 0x1000

# Создаём НОВЫЙ список
lst = lst + [4, 5]       # ← Оператор + создаёт новый объект!
print(f"ID: {id(lst)}")  # 0x2000 ← ДРУГОЙ ID!

# VS

lst = [1, 2, 3]
print(f"ID: {id(lst)}")  # 0x1000

lst.extend([4, 5])       # ← Модифицирует существующий!
print(f"ID: {id(lst)}")  # 0x1000 ← ТОТ ЖЕ ID!
```

---

## 📌 Ключевые выводы

1. **Список = Объект + Массив**
    - `id()` указывает на объект (фиксированный)
    - Массив данных может перемещаться
2. **При переполнении:**
    - ✅ Объект остаётся на месте → `id()` не меняется
    - ✗ Внутренний массив переезжает в новое место
    - Старый массив освобождается
3. **Рост списка:**
    - Изначально резерв ~1.5x
    - При переполнении новый резерв ~1.125x от текущего размера
    - Это амортизирует стоимость копирования
4. **ID меняется только:**
    - При присваивании нового объекта: `lst = [...]`
    - При операциях создающих новый список: `lst + [...]`, `lst * 2`
    - НО НЕ при: `append()`, `extend()`, `insert()`, `lst[i] = x`

## 🔍 Поиск элементов: list vs LinkedList

### ⚡ Python list — поиск по индексу O(1)

Благодаря массиву указателей фиксированного размера (8 байт каждый), Python **мгновенно** вычисляет адрес любого элемента:

```python
# Формула доступа к элементу lst[i]:
адрес_элемента = адрес_массива + i * 8  # 8 байт = размер указателя на 64-бит
```

```python
lst = [10, 20, 30, 40, 50]

# Запрос lst[3]:
#
#  ob_item (0x2000)
#       │
#       ↓
#  ┌──────┬──────┬──────┬──────┬──────┐
#  │ ptr0 │ ptr1 │ ptr2 │ ptr3 │ ptr4 │
#  │0x2000│0x2008│0x2010│0x2018│0x2020│
#  └──────┴──────┴──────┴──────┴──────┘
#                          ↑
#                  0x2000 + 3*8 = 0x2018  ← ПРЯМОЙ ПРЫЖОК!
#                          │
#                          ↓
#                        [40]
#
# Python не перебирает элементы 0, 1, 2 — он сразу прыгает на нужный!
```

```python
# Доказательство — все обращения одинаково быстры:
import timeit

lst = list(range(10_000_000))

t1 = timeit.timeit(lambda: lst[0],         number=1_000_000)
t2 = timeit.timeit(lambda: lst[5_000_000], number=1_000_000)
t3 = timeit.timeit(lambda: lst[9_999_999], number=1_000_000)

print(f"lst[0]:         {t1:.3f}s")
print(f"lst[5_000_000]: {t2:.3f}s")  # ← примерно то же время!
print(f"lst[9_999_999]: {t3:.3f}s")  # ← и это тоже!
```

### 🔗 Односвязный список — поиск по индексу O(n)

В односвязном списке каждый узел хранит **только ссылку вперёд**. Чтобы добраться до элемента `[i]`, нужно пройти всю цепочку с самого начала.

```python
# Структура узла:
┌──────────────────────────┐
│ Node                     │
│  value: 30               │
│  next:  ──────────────── │──→  следующий узел
└──────────────────────────┘

# Цепочка lst = [10, 20, 30, 40, 50]:
#
# HEAD
#  │
#  ↓
# ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐    ┌──────────┐
# │ val=10 │───→│ val=20 │───→│ val=30 │───→│ val=40 │───→│ val=50   │
# │ next ──┘    │ next ──┘    │ next ──┘    │ next ──┘    │ next=None│
# └────────┘    └────────┘    └────────┘    └────────┘    └──────────┘
#   [0]           [1]           [2]           [3]           [4]
#
# Запрос node[3]:
#   START → прыгаем на [0] → прыгаем на [1] → прыгаем на [2] → прыгаем на [3] ✓
#   Нужно 4 шага! При n=1_000_000 нужно до миллиона шагов.
```

```python
# Реализация на Python (для понимания):
class Node:
    def __init__(self, value):
        self.value = value
        self.next = None

class SinglyLinkedList:
    def __init__(self):
        self.head = None

    def get(self, index):
        current = self.head
        for _ in range(index):       # ← вот здесь O(n) шагов
            if current is None:
                raise IndexError("out of range")
            current = current.next
        return current.value

# При index=999_999 — миллион итераций вместо одного прыжка!
```

**❌ Слабые места односвязного:**
- Нет доступа к предыдущему узлу → нельзя эффективно удалить узел, зная только его
- Обход только в одну сторону → нет `lst[-1]` за O(1)

### 🔗🔗 Двусвязный список — поиск по индексу O(n), но умнее

Двусвязный список хранит ссылки **в обе стороны**. Python может оптимизировать: если `i > n//2` — идти с хвоста.

```python
# Структура узла:
┌──────────────────────────────────────┐
│ Node                                 │
│  prev: ← ссылка на предыдущий        │
│  value: 30                           │
│  next: → ссылка на следующий         │
└──────────────────────────────────────┘

# Цепочка lst = [10, 20, 30, 40, 50]:
#
# HEAD                                              TAIL
#  │                                                 │
#  ↓                                                 ↓
# ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
# │ val=10   │⇄  │ val=20   │⇄  │ val=30   │⇄  │ val=40   │⇄  │ val=50   │
# └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
#    [0]             [1]             [2]             [3]             [4]
#
# Запрос [3] из 5 элементов:
#   i=3, n=5, n//2=2  → i > n//2  → идём с ХВОСТА!
#   TAIL → [4] → [3] ✓  (всего 2 шага вместо 4)
#
# Запрос [1] из 5 элементов:
#   i=1, n=5, n//2=2  → i < n//2  → идём с ГОЛОВЫ
#   HEAD → [0] → [1] ✓  (2 шага)
```

```python
# Реализация с оптимизацией:
class DNode:
    def __init__(self, value):
        self.value = value
        self.prev = None
        self.next = None

class DoublyLinkedList:
    def __init__(self):
        self.head = None
        self.tail = None
        self.size = 0

    def get(self, index):
        if index > self.size // 2:      # ← идём с хвоста
            current = self.tail
            for _ in range(self.size - 1 - index):
                current = current.prev
        else:                           # ← идём с головы
            current = self.head
            for _ in range(index):
                current = current.next
        return current.value
```

**✅ Преимущества двусвязного:**
- Удаление узла O(1) если есть ссылка на него (знаем `prev` и `next`)
- Обход в обе стороны → `deque` в Python реализован именно так
- `lst[-1]` за O(1) через `tail`

---

### 📊 Итоговая таблица сравнения

| Операция | Python list | Односвязный | Двусвязный |
|---|---|---|---|
| Доступ по индексу `[i]` | **O(1)** ✅ | O(n) ❌ | O(n) ❌ |
| Доступ к первому `[0]` | O(1) ✅ | O(1) ✅ | O(1) ✅ |
| Доступ к последнему `[-1]` | O(1) ✅ | O(n) ❌ | O(1) ✅ |
| Поиск по значению `x in lst` | O(n) | O(n) | O(n) |
| Вставка в начало | O(n) ❌ | **O(1)** ✅ | **O(1)** ✅ |
| Вставка в конец | O(1)* ✅ | O(n)** ❌ | O(1) ✅ |
| Удаление из начала | O(n) ❌ | **O(1)** ✅ | **O(1)** ✅ |
| Удаление из середины | O(n) | O(n) | O(1)*** ✅ |
| Память на элемент | ~8 байт (указатель) | ~28+ байт (узел) | ~36+ байт (узел) |

\* амортизированное O(1) благодаря стратегии роста  
\*\* O(1) если хранить `tail`-указатель  
\*\*\* O(1) только если уже есть ссылка на узел; поиск узла по значению всё равно O(n)

### 🧠 Почему `collections.deque` — это двусвязный список?

```python
from collections import deque

d = deque([1, 2, 3, 4, 5])

# O(1) операции с обоих концов:
d.appendleft(0)   # ← вставка в начало O(1)
d.append(6)       # → вставка в конец  O(1)
d.popleft()       # ← удаление из начала O(1)
d.pop()           # → удаление из конца  O(1)

# НО: доступ по индексу — O(n)!
d[0]    # O(1) — головной узел
d[-1]   # O(1) — хвостовой узел
d[3]    # O(n) — нужно пройти 3 шага ❌
```

```python
# Когда что использовать:
#
# list    → нужен быстрый доступ по индексу, добавление в конец
# deque   → нужны быстрые операции с обоих концов (очередь, стек)
# list    → НЕ подходит для частых вставок/удалений в начале
# deque   → НЕ подходит если часто нужен lst[i] для произвольного i
```

### 🎯 Ключевой вывод про поиск

```python
# Python list:
lst[i]  →  адрес = ob_item + i * 8  →  ОДИН прыжок  →  O(1)

# Односвязный:
node[i]  →  head → [0] → [1] → ... → [i]  →  i прыжков  →  O(n)

# Двусвязный:
node[i]  →  голова или хвост → ... → [i]  →  min(i, n-i) прыжков  →  O(n)
#            но константа в 2 раза меньше, чем у односвязного
```

**Главная причина преимущества Python list:** массив указателей позволяет вычислить адрес **математически**, без обхода. Связные списки вынуждены **физически прыгать** по цепочке узлов.
