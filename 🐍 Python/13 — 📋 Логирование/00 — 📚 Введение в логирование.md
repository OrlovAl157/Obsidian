## Справка по уровням логирования

| Уровень | Номер | Использование | Примеры |
|---------|-------|----------------|---------|
| **DEBUG** | 10 | Отладка при разработке | Значения переменных, вызовы функций, SQL-запросы |
| **INFO** | 20 | Нормальная работа | Запуск приложения, действия пользователя, успешные операции |
| **WARNING** | 30 | Потенциальные проблемы | Устаревшие функции, ошибки доступа, попытки несанкционированного доступа |
| **ERROR** | 40 | Серьёзные ошибки | Необработанные исключения, ошибки валидации, сбои API |
| **CRITICAL** | 50 | Крах приложения | Невозможность подключиться к БД, переполнение памяти, отказ критичных компонентов |

## 🎯 Что такое логирование

**Логирование — это процесс записи и хранения информации о событиях и состояниях программы.**

Проще говоря, это **история жизни программы**. Вместо того чтобы выводить информацию в консоль с помощью `print()` и потом её терять, мы сохраняем всё в файл логов (лог-файл), где можно потом всё прочитать и проанализировать.

```
Программа работает
    ↓
Происходит событие (ошибка, успех, предупреждение)
    ↓
Логгер записывает это событие в файл логов
    ↓
Мы читаем логи и анализируем что произошло
```

**Логирование vs print():**

```python
# ❌ Плохо: используем print()
print("Пользователь вошёл")
print("Ошибка при загрузке файла")

# ✅ Хорошо: используем логирование
logger.info("Пользователь вошёл")
logger.error("Ошибка при загрузке файла")
```

**Почему логирование лучше:**
- ✅ История сохраняется в файл
- ✅ Разные уровни серьёзности (DEBUG, INFO, WARNING, ERROR, CRITICAL)
- ✅ Легко включить/отключить
- ✅ Форматирование и метаинформация (дата, время, имя модуля)
- ✅ Можно отправлять на разные места (экран, файл, почта, база данных)
- ✅ Профессиональный подход
- ❌ print() просто выводит в консоль и теряется

## 📊 Зачем нужно логирование

### Проблема 1: программа упала, но ты не знаешь почему

```python
# ❌ Без логирования
def load_user_data(user_id):
    file = open(f'user_{user_id}.json')  # упала? почему?
    data = json.load(file)
    return data
```

Программа упала → ты ничего не знаешь → мучаешься с отладкой.

**С логированием:**

```python
# ✅ С логированием
def load_user_data(user_id):
    logger.info(f"Попытка загрузить данные пользователя {user_id}")
    try:
        file = open(f'user_{user_id}.json')
        data = json.load(file)
        logger.info(f"Данные пользователя {user_id} успешно загружены")
        return data
    except FileNotFoundError:
        logger.error(f"Файл user_{user_id}.json не найден")
        raise
```

В логах увидишь точно что произошло!

### Проблема 2: непредвиденное поведение в продакшене

```python
# ❌ Без логирования
def process_payment(amount):
    balance = get_balance()
    if amount > balance:
        return False
    deduct_money(amount)
    return True
```

Платёж не прошёл, пользователь злится, ты не знаешь почему.

**С логированием:**

```python
# ✅ С логированием
def process_payment(amount):
    logger.info(f"Попытка снять {amount} с баланса")
    balance = get_balance()
    logger.debug(f"Текущий баланс: {balance}")
    if amount > balance:
        logger.warning(f"Недостаточно средств: требуется {amount}, доступно {balance}")
        return False
    deduct_money(amount)
    logger.info(f"Успешно снято {amount}")
    return True
```

В логах видно ровно где произошла ошибка!

### Главные причины логирования:

✅ **Отладка** — найти ошибку после краша  
✅ **Мониторинг** — следить за здоровьем приложения  
✅ **Аудит** — кто что когда делал (для безопасности)  
✅ **История** — вернуться и посмотреть что было раньше  
✅ **Analyse** — найти паттерны и проблемы

## 🔝 Иерархия уровней логирования

Уровни расположены в порядке серьёзности (от менее к более серьёзному):

```
DEBUG (10)
    ↓
INFO (20)
    ↓
WARNING (30)
    ↓
ERROR (40)
    ↓
CRITICAL (50)
```

**Важный момент:** если ты установил уровень INFO, то будут записаны события уровней INFO, WARNING, ERROR и CRITICAL. События DEBUG будут проигнорированы.

```
Если уровень = WARNING, то записываются:
    ✅ WARNING (предупреждения)
    ✅ ERROR (ошибки)
    ✅ CRITICAL (критические)
    ❌ DEBUG (отладка) — пропускаются!
    ❌ INFO (информация) — пропускаются!
```

---

## 🐛 DEBUG — отладочная информация

**Назначение:** Очень детальная информация для поиска ошибок во время разработки.

**Когда использовать:** Только во время разработки и тестирования! **НИКОГДА** в продакшене!

**Почему не в продакшене?** Потому что DEBUG может содержать конфиденциальные данные:
- 🔐 Пароли
- 🔑 API ключи
- 👤 Личные данные пользователей
- 💳 Номера кредитных карт
- 📧 Email адреса

### Примеры DEBUG событий:

```python
logger.debug("Переменная x = 42")
logger.debug("Вызов функции process_user()")
logger.debug("SELECT * FROM users WHERE id = 123")  # ❌ опасно! может содержать данные
logger.debug(f"API ответ: {response}")
logger.debug("Вход в функцию calculate()")
```

### Реальный пример:

```python
def authenticate(username, password):
    logger.debug(f"Попытка входа: {username}")  # ✅ нормально
    logger.debug(f"Пароль: {password}")          # ❌ опасно! не логируй пароли!
    
    if verify_password(username, password):
        logger.debug(f"Пароль верный для {username}")  # ✅ можно
        return True
    
    logger.debug(f"Ошибка пароля для {username}")      # ✅ нормально
    return False
```

**✅ Главные правила DEBUG:**

✅ Используй только в разработке  
✅ Логируй значения переменных  
✅ Логируй вызовы функций  
✅ Логируй SQL-запросы (но без данных!)  
✅ **НЕ логируй пароли, ключи, персональные данные!**

## ℹ️ INFO — информация о работе

**Назначение:** Информация о нормальной работе приложения. Это уровень по умолчанию для мониторинга.

**Когда использовать:** На всех этапах — разработка, тестирование, продакшен.

**Что логировать:**

✅ Запуск и остановка приложения/компонентов  
✅ Вход и выход из системы пользователей  
✅ Успешное выполнение важных задач  
✅ Значимые действия пользователя  
✅ Подключение к базам данных  
✅ Загрузка конфигурации  

### Примеры INFO событий:

```python
logger.info("Приложение запущено")
logger.info(f"Пользователь {username} вошёл в систему")
logger.info(f"Обрабатываю заказ #{order_id}")
logger.info(f"Данные успешно сохранены в базу (записей: {count})")
logger.info("Соединение с БД установлено")
logger.info(f"Пользователь {username} вышел из системы")
```

### Реальный пример:

```python
def process_order(order_id):
    logger.info(f"Начало обработки заказа {order_id}")
    
    order = fetch_order(order_id)
    logger.info(f"Заказ {order_id} получен")
    
    payment = process_payment(order.amount)
    logger.info(f"Платёж за заказ {order_id} обработан")
    
    send_notification(order.customer_email)
    logger.info(f"Уведомление отправлено клиенту {order.customer_email}")
    
    logger.info(f"Заказ {order_id} успешно обработан")
    return True
```

**✅ Главные правила INFO:**

✅ Логируй важные события  
✅ Логируй начало и конец больших операций  
✅ Логируй успешные действия  
✅ **НЕ переполняй логи** — логируй только действительно важные события  
✅ Используй во всех окружениях

## ⚠️ WARNING — предупреждение

**Назначение:** Что-то странное, но приложение работает. Требует внимания, но не срочно.

**Когда использовать:** Когда произошло что-то неожиданное, но программа может продолжить работу.

**Что логировать:**

✅ Использование устаревших функций  
✅ Ошибки доступа к конфигам (но используются значения по умолчанию)  
✅ Внешний API отвечает медленно  
✅ Возможные попытки несанкционированного доступа  
✅ Неполные данные (но обработка продолжается)  
✅ Ресурсное предупреждение (память, место на диске)  

### Примеры WARNING событий:

```python
logger.warning("Используешь устаревшую функцию process_user()")
logger.warning("Конфиг не найден, используются значения по умолчанию")
logger.warning("API отвечает медленно (ответ > 5 сек)")
logger.warning(f"Много неудачных попыток входа для пользователя {username}")
logger.warning(f"Осталось менее 10% памяти на диске")
logger.warning("Email не смог доставить письмо, повтор позже")
```

### Реальный пример:

```python
def fetch_external_api(url, timeout=5):
    start = time.time()
    try:
        response = requests.get(url, timeout=timeout)
        elapsed = time.time() - start
        
        if elapsed > 3:
            logger.warning(f"API {url} ответил медленно ({elapsed:.2f} сек)")
        
        return response
    except requests.Timeout:
        logger.warning(f"API {url} не ответил за {timeout} сек, повторяю запрос")
        return retry_api(url)
```

**✅ Главные правила WARNING:**

✅ Логируй неожиданные, но некритичные ситуации  
✅ Приложение работает, но нужна внимание  
✅ Обычно требует расследования в ближайшем будущем  
✅ Не критично для работы, но может привести к проблемам

## ❌ ERROR — ошибка

**Назначение:** Серьёзная ошибка, которая помешала выполнению операции. Приложение продолжает работать, но эта операция не выполнена.

**Когда использовать:** Когда произошла ошибка, которую нужно срочно исправить.

**Что логировать:**

✅ Необработанные исключения  
✅ Ошибки валидации данных  
✅ Ошибки при работе с файлами  
✅ Ошибки десериализации (JSON, XML, и т.д.)  
✅ Сбои при работе с внешними API  
✅ Ошибки базы данных (кроме подключения)  

### Примеры ERROR событий:

```python
logger.error(f"Исключение при обработке заказа {order_id}: {e}")
logger.error("JSON невалидный, не смогу распарсить")
logger.error(f"Ошибка базы данных: {db_error}")
logger.error(f"Не смог сохранить файл {filename}: {e}")
logger.error(f"API вернул ошибку: {status_code} {message}")
```

### Реальный пример:

```python
def save_user_data(user_id, data):
    try:
        validate_data(data)
        db.insert('users', data)
        logger.info(f"Пользователь {user_id} сохранён")
    except ValidationError as e:
        logger.error(f"Ошибка валидации для пользователя {user_id}: {e}")
        return False
    except DatabaseError as e:
        logger.error(f"Ошибка БД при сохранении пользователя {user_id}: {e}")
        return False
```

**✅ Главные правила ERROR:**

✅ Логируй когда операция не выполнена  
✅ Всегда включай сообщение об ошибке  
✅ Требует немедленного внимания  
✅ Приложение работает, но нужна исправить код или данные  
✅ На продакшене это мониторится

## 🔴 CRITICAL — критическая ошибка

**Назначение:** Краш приложения. Что-то пошло **очень** плохо, приложение не может продолжать работу.

**Когда использовать:** Только когда всё совсем плохо и приложение не может работать.

**Что логировать:**

✅ Невозможно подключиться к БД (критическая зависимость)  
✅ Ошибка при запуске приложения  
✅ Переполнение памяти  
✅ Отказ критически важного компонента  
✅ Нарушение целостности данных  

### Примеры CRITICAL событий:

```python
logger.critical("Не смогу подключиться к БД, приложение неработоспособно")
logger.critical("Ошибка при инициализации приложения, выход")
logger.critical("Детектирована попытка несанкционированного доступа, выключаю систему")
logger.critical("Память переполнена, выход из приложения")
logger.critical("Критичный компонент упал, невозможно продолжать")
```

### Реальный пример:

```python
def initialize_app():
    logger.info("Инициализация приложения")
    
    try:
        config = load_config()
        logger.info("Конфиг загружен")
    except Exception as e:
        logger.critical(f"Ошибка при загрузке конфига: {e}, приложение не запустится")
        sys.exit(1)
    
    try:
        db.connect(config.db_url)
        logger.info("Соединение с БД установлено")
    except Exception as e:
        logger.critical(f"Не смогу подключиться к БД {config.db_url}: {e}")
        sys.exit(1)
    
    logger.info("Приложение успешно инициализировано")
```

**✅ Главные правила CRITICAL:**

✅ Логируй **ДО завершения приложения**  
✅ Часто означает `sys.exit()`  
✅ Требует немедленного исправления и перезагрузки  
✅ На продакшене это срабатывает алерт (SMS, звонок, и т.д.)  
✅ Используй редко, только для реальных крахов

## 📝 Примечания и особенности

### Примечание 1: TRACE уровень

В некоторых языках (Java, C#) есть уровень **TRACE** ещё ниже DEBUG. Это самая детальная информация. В Python это не стандартно, но можно сделать самому.

### Примечание 2: Альтернативные имена

В некоторой документации встречаются:
- **WARN** вместо WARNING (идентично)
- **FATAL** вместо CRITICAL (идентично)

Python поддерживает оба названия.

### Примечание 3: Русское название

Логирование иногда называют **журналированием**. Это одно и то же.

### Примечание 4: Уровень по умолчанию

По умолчанию логируется **WARNING и выше**. Поэтому DEBUG и INFO требуют явной настройки.

## ✅ Главные правила

✅ DEBUG — **только разработка**, содержит детали, **NO конфиденциальные данные**  
✅ INFO — **нормальная работа**, запуск/остановка, успехи  
✅ WARNING — **неожиданно**, но работает, требует внимания  
✅ ERROR — **ошибка операции**, приложение работает, но действие не выполнено  
✅ CRITICAL — **крах приложения**, выход неизбежен  
✅ Всегда используй уровни в порядке серьёзности  
✅ **НИКОГДА не логируй пароли, ключи, персональные данные в DEBUG!**  
✅ Используй логирование вместо print()  
✅ Логирование — это **история жизни программы**
