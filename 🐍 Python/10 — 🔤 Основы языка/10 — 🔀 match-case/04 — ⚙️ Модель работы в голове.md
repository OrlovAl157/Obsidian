## 🎯 Главная идея

Большинство новичков думают так:

```text
match-case = большой switch
```

Но это не совсем так. Лучше думать так:

```text
match-case = попытка разобрать объект по шаблону
```

Python не спрашивает: *"Равно ли это значение?"*

Он спрашивает: *"Подходит ли объект под этот шаблон? Можно ли извлечь из него данные?"*

Это принципиально разные модели мышления. Первая (switch) работает только для литералов. Вторая (разбор по шаблону) объясняет вообще всё поведение `match-case`, включая случаи, где никакого сравнения не происходит вовсе.

---

## 🔢 Общий алгоритм

Для конструкции:

```python
match subject:
    case pattern if guard:
        ...
```

Python работает так:

```text
1. Вычислить subject (ОДИН раз, не на каждый case)

2. Взять первый case по порядку

3. Попытаться сопоставить subject с pattern
   (это проверка ФОРМЫ, не значения как такового)

4. Если шаблон содержит имена —
   создать переменные и связать их со значениями
   (это происходит ДО guard)

5. Если есть guard (if) —
   проверить условие

6. Если всё успешно —
   выполнить тело case
   и выйти из match (без break, как и так)

7. Иначе перейти к следующему case
   (созданные на шаге 4 имена уже могут существовать в области видимости)
```

**Важное:** каждый `case` проверяется заново по этому же алгоритму, независимо от соседних. Нет общего состояния между ними, кроме общего `subject`.

---

## ❗ Самая важная мысль

```text
case не всегда означает сравнение
```

Например:

```python
match 7:
    case length:
        ...
```

Многие думают, что здесь происходит `7 == length`. Это неверно.

Здесь происходит:

```python
length = 7
```

`case length` — это не сравнение. Это **захват значения** (capture pattern). Одиночное имя в шаблоне всегда совпадает с чем угодно и просто забирает значение целиком.

---

## 🟢 Capture pattern — `case length`

```python
match len(s):
    case length if length <= 5:
        print(length)
```

Если `len(s) == 7`, Python делает:

```text
1. subject = 7

2. case length
   ↓
   length = 7        # захват, не сравнение

3. Проверка guard:
   length <= 5
   ↓
   False

4. Переходим к следующему case
```

`length` здесь **не сравнивается** — он **создаётся**. Вся реальная логика проверки лежит в `guard`, не в самом шаблоне.

---

## 🔵 Literal pattern — `case 5`

Настоящее сравнение выглядит так:

```python
match 7:
    case 5:
        ...
```

Тут происходит:

```python
7 == 5
```

Результат — `False`. Здесь никаких переменных не создаётся, потому что `5` — это не имя, а конкретное значение. Шаблон-литерал — единственный случай, где модель "сравнение" действительно работает буквально.

---

## 📋 Sequence pattern — `case [x, y]`

```python
match [10, 20]:
    case [x, y]:
        ...
```

Алгоритм:

```text
1. Это последовательность?
   Да

2. Длина равна 2?
   Да

3. x = 10

4. y = 20

5. Совпадение успешно
```

Результат: `x == 10`, `y == 20`.

**Важное:** проверка формы (тип + длина) происходит **раньше**, чем создание имён. Сначала Python убеждается, что объект вообще можно "разобрать" таким образом, и только потом распаковывает.

---

## ❌ Почему не работает `case [x]`

```python
match [10, 20]:
    case [x]:
        ...
```

Алгоритм:

```text
1. Это последовательность?
   Да

2. Длина равна 1?
   Нет (длина 2)
```

Паттерн провалился. **До создания `x` дело даже не дошло** — переменная не создаётся вообще, потому что проверка формы (длина) идёт раньше любого захвата.

Это ключевое отличие от обычного присваивания: в `x = [10, 20][0]` `x` создалась бы всегда. В `case [x]` — только если форма точно совпала.

## 🗂 Mapping pattern — `case {key: city}`

```python
years = {
    1886: 'London',
    1998: 'Raccoon City',
    2077: 'Night City'
}

match years:
    case {1886: city}:
        print(city)
```

Многие ошибочно думают, что здесь происходит `years == {1886: city}`. Но это **не сравнение словарей**.

Алгоритм такой:

```text
1. Это словарь (Mapping)?
   Да

2. Есть ключ 1886?
   Да

3. city = years[1886]

4. Совпадение успешно
```

Получаем `city = 'London'`.

---

## ❓ Почему лишние ключи не мешают

`case {1886: city}` означает:

```text
Найди словарь,
в котором есть ключ 1886.
```

А не:

```text
Найди словарь,
который состоит ТОЛЬКО из этого ключа.
```

Поэтому словарь с тремя ключами (`1886`, `1998`, `2077`) всё равно успешно подходит — шаблон ничего не утверждает про остальные ключи, он проверяет только наличие нужного.

Это прямая противоположность поведению sequence pattern, где длина должна совпасть **точно**.

---

## 🗂 Mapping с несколькими ключами

```python
match years:
    case {
        1886: city,
        1998: place
    }:
        ...
```

Алгоритм:

```text
Есть ключ 1886?
Да

Есть ключ 1998?
Да

city = years[1886]
place = years[1998]
```

Успех. Каждый ключ в шаблоне проверяется отдельно через `in`, и только если **все** перечисленные ключи присутствуют — шаблон совпадает целиком.

---

## 🔶 Порядок: захват потом guard

Важно понимать порядок операций.

Для:

```python
case length if length <= 5:
```

происходит:

```text
1. length = subject     # сначала захват

2. Проверить:
   length <= 5          # потом guard

3. Если True —
   выполнить тело

4. Если False —
   перейти к следующему case
```

То есть:

```text
СНАЧАЛА захват
ПОТОМ guard
```

Имя уже существует в момент проверки `guard`, поэтому в самом условии `if` можно свободно использовать только что захваченные переменные.

---

## 📐 Формула для запоминания

```text
match-case

=

проверка шаблона (формы)
+
извлечение данных
+
создание переменных
+
guard (опционально)
```

А **НЕ**:

```text
match-case = большое сравнение if/elif
```

Это главная ошибка при изучении `match-case`. Switch-модель работает только для частного случая литералов — она ломается на capture pattern, sequence pattern и mapping pattern, где идёт не сравнение значений, а проверка формы объекта с последующим извлечением данных.
