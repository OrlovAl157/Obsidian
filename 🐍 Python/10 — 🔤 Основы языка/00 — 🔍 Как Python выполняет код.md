# 🐍 Как Python выполняет код

## Полная цепочка

```
твой .py файл
      ↓  (компилятор Python)
байткод (.pyc)
      ↓  (PVM интерпретирует байткод)
вызов C-функций CPython
      ↓  (уже скомпилированный C → машинный код)
   процессор
```

---

## Что такое байткод

Байткод — это **промежуточное представление** кода. Не машинный код, но ближе к нему чем текст.

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
# LOAD_FAST   0 (a)
# LOAD_FAST   1 (b)
# BINARY_OP   0 (+)
# RETURN_VALUE
```

Байткод **не компилируется напрямую в машинный код** — он интерпретируется PVM.  
Но каждая инструкция байткода **вызывает машинный код** внутри CPython.

```
байткод → (интерпретация PVM) → выполнение машинного кода
```

> [!tip] Кэширование
> Python сохраняет байткод в `.pyc` файлы — если код не менялся, парсинг `.py` пропускается.

---

## Что такое PVM

PVM (Python Virtual Machine) — это **диспетчер выполнения**. Большой цикл на C внутри `ceval.c`:

```c
while (running) {
    opcode = next_instruction();
    switch(opcode) {
        case LOAD_FAST:    // C-код...
        case BINARY_OP:    // C-код...
        case RETURN_VALUE: // C-код...
    }
}
```

PVM:
1. Читает инструкцию байткода
2. Решает что делать (диспетчеризация)
3. Вызывает нужную C-функцию

> PVM **не переводит** байткод в машинный код.  
> PVM — это **диспетчер**: она сама уже является машинным кодом (скомпилирована из C заранее) и вызывает другие C-функции.

---

## Путь простого сложения a + b

```
байткод: BINARY_OP
      ↓
PVM вызывает PyNumber_Add(a, b)
      ↓
PyNumber_Add смотрит на тип объекта
      ↓
вызывает PyLong_Add (для int)
      ↓
извлекает числа из структуры PyObject
      ↓
C-сложение → новый PyObject с результатом
```

Даже `1 + 2` — это не одна инструкция, а целая цепочка C-вызовов.

## Объекты Python — структуры в C

Каждый объект Python — это структура `PyObject` в C (упрощённо):

```c
typedef struct {
    int ob_refcnt;          // счётчик ссылок (сборщик мусора)
    PyTypeObject *ob_type;  // указатель на тип объекта
    // данные зависят от типа
} PyObject;
```

> [!note] PyLongObject на самом деле
> В реальности `PyLongObject` поддерживает **произвольно большие числа** — там массив "цифр", а не один `long`. Именно поэтому `int` в Python никогда не переполняется (в отличие от C).

## Где происходит преобразование в 01

**Один раз** — при сборке CPython разработчиками:

```
разработчики пишут CPython на C
      ↓  (gcc/clang компилирует)
python.exe — это и есть 01
```

Твой Python-код **не компилируется** в машинный код,  
но он **постоянно вызывает** машинный код через C-функции CPython.

---

## Главная идея

> Python-код компилируется в байткод, который интерпретируется виртуальной машиной CPython.  
> PVM управляет выполнением и вызывает низкоуровневые C-функции, которые уже исполняются как машинный код.

**Python — язык поверх C.** Именно поэтому Python медленнее C — лишний слой интерпретации.  
Именно поэтому `numpy` быстрый — критичные части написаны на C напрямую.

---

## Почему Python медленнее C

| | C | Python |
|---|---|---|
| Компиляция | один раз → 01 | байткод → PVM → C-функции |
| Простое сложение | 1 инструкция | цепочка C-вызовов |
| Тип переменной | известен на этапе компиляции | проверяется в рантайме |
| Объект в памяти | просто число | структура PyObject с метаданными |

---

## Исключения — когда Python компилирует в 01

| Инструмент | Что делает |
|---|---|
| **PyPy** | полноценный JIT — компилирует горячий код в машинный код в рантайме |
| **Numba** | JIT для отдельных функций (особенно числодробилки) |
| **CPython 3.11+** | adaptive interpreter — не JIT, но оптимизирует горячие инструкции |

## Зачем нужен байткод

1. **Скорость** — не нужно каждый раз парсить текст `.py`
2. **Простота для PVM** — чёткие однозначные команды вместо текста
3. **Платформонезависимость** — байткод одинаков на Windows/Linux/Mac
4. **Оптимизации** — компилятор упрощает код до байткода:
   ```python
   x = 2 * 8   # в байткоде сразу LOAD_CONST 16
   ```
