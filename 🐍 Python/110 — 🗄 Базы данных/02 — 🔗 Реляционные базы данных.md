# 🔗 Реляционные базы данных

## 📌 Коротко

> Реляционная БД — набор связанных таблиц, где данные организованы в строки и столбцы, а связи между таблицами устанавливаются через ключи.

---

## 📖 Теория

### История

Реляционная модель разработана в **1970 году Эдгаром Коддом** — математиком и специалистом в области информатики. В 1981 году он получил премию Тьюринга за вклад в теорию баз данных.

Название **«реляционная»** — от латинского *relation* (отношение). В теории множеств таблица называется **отношением**.

---

### Структура таблицы

| Термин | Что это |
|---|---|
| **Таблица** | Физическое представление отношения |
| **Запись (строка)** | Набор значений об одной сущности |
| **Поле (столбец)** | Отдельная характеристика сущности |
| **Значение** | Конкретные данные на пересечении строки и столбца |

```
Таблица: users
┌────┬──────────┬──────────┬──────────────┐
│ ID │ Имя      │ Фамилия  │ Дата рожд.   │  ← поля (столбцы)
├────┼──────────┼──────────┼──────────────┤
│  1 │ Билл     │ Гейтс    │ 28.10.1955   │  ← запись (строка)
│  2 │ Алан     │ Тьюринг  │ 23.06.1912   │
│  3 │ Эдгар    │ Кодд     │ 19.08.1923   │
└────┴──────────┴──────────┴──────────────┘
```

### Свойства таблицы в реляционной БД

- Каждое поле имеет **уникальное имя** и **тип данных** (текст, число, дата)
- Каждая таблица имеет **уникальное имя** в рамках БД
- Строки не имеют гарантированного порядка без явной сортировки

---

### Первичный ключ (Primary Key)

**Первичный ключ** — поле (или несколько полей), значения которого **однозначно идентифицируют** каждую строку таблицы.

Требования:
- ✅ Уникальный — значения не повторяются
- ✅ Не NULL — всегда заполнен

**Простой ключ** — состоит из одного поля:
```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT
);
```

**Составной ключ** — состоит из нескольких полей:
```sql
CREATE TABLE enrollments (
    student_id INTEGER,
    course_id  INTEGER,
    PRIMARY KEY (student_id, course_id)
);
```

> ⚠️ Первичные ключи не обязательны технически, но большинство разработчиков создают их для каждой таблицы.

**Суррогатный vs естественный ключ:**

| Тип | Описание | Пример |
|---|---|---|
| **Суррогатный** | Искусственный `id`, автоинкремент | `1, 2, 3, 4...` |
| **Естественный** | Поле из реальных данных | email, паспорт, ИНН |

> 💡 На практике почти всегда используют суррогатный `id` — естественные ключи могут меняться.

---

### Внешний ключ (Foreign Key)

**Внешний ключ** — поле в одной таблице, которое ссылается на **первичный ключ другой таблицы**.

```
users (родительская таблица):        orders (дочерняя таблица):
┌────┬──────────┐                    ┌────┬─────────┬──────────┐
│ id │ name     │                    │ id │ user_id │ product  │
├────┼──────────┤                    ├────┼─────────┼──────────┤
│  1 │ Саша     │◄───────────────────│  1 │    1    │ Ноутбук  │
│  2 │ Иван     │◄──────────┐        │  2 │    1    │ Мышка    │
└────┴──────────┘           └────────│  3 │    2    │ Книга    │
                                     └────┴─────────┴──────────┘
                                           ↑
                                      внешний ключ
```

- **Родительская таблица** — та, на которую ссылаются
- **Дочерняя таблица** — та, которая содержит внешний ключ

> 💡 Внешний ключ может состоять из одного или нескольких полей — так же как первичный.

---

### Схема данных

**Схема данных** — группа связанных таблиц, изображённая в виде диаграммы с линиями/стрелками между таблицами. В схеме указываются имена таблиц и их поля — без содержимого.

---

## 💡 Полезные советы

- **Гибкость реляционных БД** — таблицы в значительной степени независимы. Можно добавлять, удалять и изменять данные в одной таблице, не затрагивая другие.
- **Реляционная алгебра** — математическая основа модели. В ней: поле = *атрибут*, строка = *кортеж*, набор возможных значений поля = *домен*.
