# 🔗 Callable, Protocol и структурная типизация

> [!info] Коротко
> Эта тема начинается с вопроса "как типизировать функцию, которую передают как аргумент", а заканчивается гораздо более широкой идеей: в Python важен не только номинальный тип объекта, но и то, что он умеет делать. Отсюда растут `Callable`, `Protocol`, `collections.abc` и различие между номинальной и структурной типизацией.

## Справка по теме

| Что | Пример | Смысл |
|---|---|---|
| Функция без аргументов | `Callable[[], str]` | вызываемый объект, возвращающий строку |
| Функция с двумя аргументами | `Callable[[int, int], int]` | принимает два `int`, возвращает `int` |
| Любая сигнатура | `Callable[..., object]` | вызываемый объект без точной сигнатуры |
| Протокол | `class Drawable(Protocol): ...` | важна структура поведения |
| Интерфейс коллекции | `Iterable[str]` | важна возможность итерироваться |

## Callable

`Callable` нужен, когда функция принимает другую функцию или возвращает ее.

```python
from collections.abc import Callable


def apply(a: int, b: int, op: Callable[[int, int], int]) -> int:
    return op(a, b)
```

Здесь `op` — любой вызываемый объект, который принимает два `int` и возвращает `int`.

Пример использования:

```python
result = apply(2, 3, lambda x, y: x + y)
```

Есть и другие формы:

```python
Callable[[], str]
Callable[[str], bool]
Callable[..., int]
```

Последний вариант с `...` полезен, когда функция вызываемая, но точную сигнатуру мы сейчас не описываем.

## Когда одного Callable уже мало

`Callable` хорош, когда сигнатура простая. Но у него есть ограничение: он умеет выразить список аргументов и возвращаемый тип, но не описывает более богатый контракт поведения.

Например, если нам нужен объект, у которого есть метод `render()`, одного `Callable` уже недостаточно. Или если объект должен поддерживать и вызов, и дополнительные методы.

Тогда на сцену выходит `Protocol`.

---

## Номинальная и структурная типизация

Номинальная типизация смотрит на объявленный тип и наследование. Если класс не наследуется от нужной базы, формально он "не тот".

Структурная типизация смотрит на поведение. Если объект имеет нужные методы и свойства, он подходит, даже без явного наследования.

Упрощенно:

- номинальная: "кем ты объявлен";
- структурная: "что ты умеешь делать".

Python по духу очень близок к идее "утиная типизация": если объект выглядит и ведет себя как нужный интерфейс, его можно использовать. `Protocol` формализует это для типизации.

## Protocol

`Protocol` позволяет описать не родословную класса, а его форму.

```python
from typing import Protocol


class Drawable(Protocol):
    def draw(self) -> str:
        ...
```

Теперь любой объект с методом `draw()` подходящего типа будет соответствовать этому протоколу:

```python
class Circle:
    def draw(self) -> str:
        return "circle"


class Square:
    def draw(self) -> str:
        return "square"


def render(obj: Drawable) -> None:
    print(obj.draw())
```

Ни `Circle`, ни `Square` не обязаны наследоваться от `Drawable`. Это и есть главное удобство структурной типизации.

Такой подход отлично подходит для:

- плагинов;
- адаптеров;
- объектов с общим поведением, но разной иерархией;
- библиотечного кода, который хочет быть гибким.

> [!tip] Практический смысл
> `Protocol` полезен там, где важен контракт поведения, а не принадлежность к конкретной базе.

## Supports-* протоколы

В `typing` уже есть готовые маленькие протоколы вида `SupportsInt`, `SupportsFloat`, `SupportsBytes` и другие. Они выражают идею: объект поддерживает определенную операцию.

Например, если функция может принимать не только `int`, но любой объект, который можно преобразовать в целое число, логически это уже не просто `int`, а "что-то, поддерживающее `__int__()`".

Это полезно знать как концепт: типизация в Python умеет быть не только про классы, но и про свойства поведения.

## Абстрактные классы и collections.abc

Рядом с `Protocol` есть еще слой абстрактных классов и интерфейсов из `collections.abc`:

```python
from collections.abc import Iterable, Sequence


def show_all(items: Iterable[str]) -> None:
    for item in items:
        print(item)


def first_two(items: Sequence[str]) -> tuple[str, str]:
    return items[0], items[1]
```

`Iterable[str]` говорит: объект можно перебирать.

`Sequence[str]` говорит больше: объект поддерживает индексирование и длину, как последовательность.

Это удобнее и точнее, чем всегда писать `list[str]`, если конкретная реализация нам не важна.

Абстрактные классы и протоколы иногда пересекаются по смыслу, но интуитивно можно запомнить так:

- `collections.abc` часто описывает стандартные интерфейсы коллекций и вызова;
- `Protocol` удобен для своих пользовательских контрактов.

---

## Главные правила

- `Callable` хорош для простых случаев "функция как аргумент" или "функция как результат".
- Если важны методы и поведение объекта, а не его наследование, смотри в сторону `Protocol`.
- Для коллекций часто полезнее `Iterable`, `Sequence`, `Mapping`, чем жесткая привязка к `list` и `dict`.
- Структурная типизация делает Python-код гибче и ближе к реальной утино-типизированной практике.
- Чем яснее тип выражает поведение, тем меньше приходится объяснять код комментариями.
