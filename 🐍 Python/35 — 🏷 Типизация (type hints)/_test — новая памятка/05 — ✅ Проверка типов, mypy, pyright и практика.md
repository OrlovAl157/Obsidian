# ✅ Проверка типов: mypy, pyright и практический подход

> [!info] Коротко
> Аннотации начинают по-настоящему окупаться, когда их проверяет инструмент. В обычном Python типы сами по себе не гарантируют безопасность. `mypy` и `pyright` превращают подсказки в реальные сигналы об ошибках еще до запуска программы.

## Зачем нужен type checker

Аннотация без проверки — это в лучшем случае документация. Type checker делает из нее инструмент обратной связи.

Пример:

```python
def greet(name: str) -> str:
    return f"Hello, {name}"

greet(10)
```

Python сам по себе может не пожаловаться здесь сразу так, как мы ожидаем от строгой типизации. Но `mypy` и `pyright` увидят несоответствие еще до запуска.

Они особенно полезны для:

- больших функций с ветвлениями;
- nullable-значений;
- словарей и сложных структур;
- рефакторинга;
- декораторов и generic-кода.

---

## mypy

`mypy` — самый известный type checker в мире Python.

Базовое использование:

```bash
pip install mypy
mypy file.py
mypy .
```

Более строгий режим:

```bash
mypy --strict .
```

Также часто полезны настройки через `mypy.ini` или `pyproject.toml`. Типичный путь — начинать мягче, а потом постепенно ужесточать правила.

Пример полезных настроек:

```ini
[mypy]
python_version = 3.11
warn_return_any = True
warn_unused_configs = True
disallow_untyped_defs = True
```

---

## pyright

`pyright` часто воспринимают как более быстрый и очень приятный инструмент, особенно в связке с редактором.

Базовое использование:

```bash
pip install pyright
pyright
pyright file.py
```

Он особенно удобен там, где хочется живой обратной связи прямо в IDE.

На практике выбор между `mypy` и `pyright` часто не вопрос "какой истинно правильный", а вопрос среды, привычки и конкретного проекта.

## Какие ошибки они реально ловят

Самые частые полезные сигналы:

### Неправильный тип аргумента

```python
def square(x: int) -> int:
    return x * x

square("4")
```

### Потенциальный `None`

```python
def find_user(name: str) -> str | None:
    ...

user = find_user("Ann")
print(user.upper())
```

Type checker скажет: сначала проверь, что `user` не `None`.

### Неправильный тип внутри коллекции

```python
values: list[int] = [1, 2, "3"]
```

### Нарушение контракта `TypedDict`

Если поле пропущено, названо неверно или имеет неправильный тип, checker это увидит.

## Type narrowing

Одна из важнейших идей в практике типизации — сужение типа.

Если у нас:

```python
value: str | None
```

то после проверки:

```python
if value is not None:
    print(value.upper())
```

анализатор понимает, что внутри блока `if` тип уже `str`.

То же работает с `isinstance()` и другими проверками ветвления. Это превращает типизацию из "мертвых подписей" в живую логику программы.

---

## Постепенное внедрение

Не нужно типизировать все сразу и идеально. Нормальная стратегия выглядит так:

1. Сначала аннотировать сигнатуры функций.
2. Затем описать основные коллекции и возвращаемые значения.
3. Потом закрыть `Optional`, сложные словари и важные модели.
4. После этого подключить checker в проект или хотя бы прогонять вручную.
5. Уже потом ужесточать режим.

Так путь получается реалистичным, а не героически болезненным.

> [!tip] Рабочий подход
> Лучше 70% понятной типизации, которая реально проверяется, чем 100% красивых аннотаций без прогонов checker-а.

## Когда допустим type: ignore

`# type: ignore` — не зло, но и не универсальная таблетка.

Он уместен, когда:

- библиотека плохо типизирована;
- checker ошибается на пограничном случае;
- ты осознанно обходишь ограничение и понимаешь, почему.

Плохой стиль — сыпать `type: ignore` везде, где лень разобраться. Тогда доверие к типизации быстро разваливается.

Если такой комментарий появился, хорошо бы понимать причину и по возможности оставить рядом короткое пояснение.

---

## Практический стиль типизации

Самый полезный стиль обычно такой:

- не типизировать ради галочки все подряд;
- держать сигнатуры чистыми и читаемыми;
- сложные типы выносить в псевдонимы;
- на границах системы явно обрабатывать `None`, JSON-подобные данные и внешние зависимости;
- регулярно прогонять checker, а не хранить типы как декоративный слой.

Если типовая запись мешает чтению, это сигнал не только к сокращению типов, но иногда и к упрощению API.

---

## Главные правила

- Без `mypy` или `pyright` аннотации дают только часть пользы.
- Главные реальные находки checker-а: `None`, неверные аргументы, сломанные коллекции, несоответствия контракту.
- Внедрять типизацию лучше постепенно, а не залпом.
- `type: ignore` допустим как исключение, а не как фон.
- Лучшая типизация — та, которую команда понимает и действительно проверяет.
