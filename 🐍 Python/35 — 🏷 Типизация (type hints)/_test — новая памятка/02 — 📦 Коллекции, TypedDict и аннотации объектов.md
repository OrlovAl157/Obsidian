# 📦 Коллекции, TypedDict и аннотации объектов

> [!info] Коротко
> Типизация коллекций — это момент, где аннотации становятся по-настоящему полезными. Именно здесь мы перестаем писать просто `list` и начинаем явно описывать структуру данных: какие элементы лежат в списке, какие ключи у словаря, какие поля ожидаются в объекте-подобной записи.

## Справка по теме

| Что | Пример | Что означает |
|---|---|---|
| Список строк | `list[str]` | список, где каждый элемент строка |
| Кортеж координат | `tuple[int, int]` | два числа в фиксированном порядке |
| Множество тегов | `set[str]` | набор уникальных строк |
| Словарь возрастов | `dict[str, int]` | строковый ключ и числовое значение |
| Список словарей | `list[dict[str, str]]` | вложенная структура |
| TypedDict | `class UserData(TypedDict): ...` | словарь с фиксированными полями |

## Типизация стандартных коллекций

Начальный уровень выглядит так:

```python
names: list[str] = ["Ann", "Bob"]
scores: list[int] = [10, 20, 30]
point: tuple[int, int] = (5, 7)
tags: set[str] = {"python", "typing"}
ages: dict[str, int] = {"Ann": 20, "Bob": 22}
```

Самое важное здесь — тип элементов.

Плохой вариант:

```python
items: list = [1, 2, 3]
```

Формально это аннотация, но почти бесполезная: из нее не видно, что ожидается внутри списка.

Хороший вариант:

```python
items: list[int] = [1, 2, 3]
```

То же самое касается словаря. `dict` без ключа и значения обычно слишком расплывчат.

## Вложенные структуры

Чем сложнее данные, тем ценнее аннотации. Например:

```python
users: list[dict[str, str]] = [
    {"name": "Ann", "email": "ann@example.com"},
    {"name": "Bob", "email": "bob@example.com"},
]
```

Или:

```python
stats: dict[str, list[int]] = {
    "math": [5, 4, 5],
    "english": [4, 5, 5],
}
```

Такие аннотации уже работают как компактная схема данных. Даже без чтения тела функции видно, что за структура проходит через код.

Если тип становится очень длинным, полезно вынести его в псевдоним:

```python
from typing import TypeAlias

UserRecord: TypeAlias = dict[str, str]
UserList: TypeAlias = list[UserRecord]
```

## Когда обычного dict уже мало

Обычный `dict[str, str]` или `dict[str, object]` хорош, пока у тебя простые данные или временная структура. Но когда словарь начинает играть роль "объекта" с известными полями, обычной аннотации уже мало.

Например:

```python
user: dict[str, str] = {
    "name": "Ann",
    "email": "ann@example.com",
}
```

Из аннотации видно только то, что ключи и значения строки. Но не видно:

- какие ключи обязательны;
- можно ли добавлять другие;
- какое поле опционально;
- есть ли разные типы у разных полей.

Здесь как раз нужен `TypedDict`.

---

## TypedDict

`TypedDict` описывает словарь с фиксированными полями.

```python
from typing import TypedDict


class UserData(TypedDict):
    name: str
    email: str
    age: int
```

Теперь можно писать так:

```python
user: UserData = {
    "name": "Ann",
    "email": "ann@example.com",
    "age": 20,
}
```

Преимущество огромное: type checker понимает структуру ключей и их типы по отдельности.

Это особенно полезно для:

- JSON-подобных данных;
- словарей конфигурации;
- результатов API;
- временных структур между слоями программы.

Если пытаться хранить все это через `dict[str, object]`, код быстро становится мутным.

> [!tip] Когда выбирать
> Если словарь ведет себя как запись с постоянными полями, почти всегда стоит подумать о `TypedDict`.

## NotRequired, Required и total

Не все поля обязаны присутствовать всегда. Для этого у `TypedDict` есть дополнительные механизмы.

Пример с необязательным полем:

```python
from typing import NotRequired, TypedDict


class UserData(TypedDict):
    name: str
    email: str
    phone: NotRequired[str]
```

Теперь `phone` можно не указывать.

Есть и обратная ситуация: словарь по умолчанию "неполный", но отдельные поля обязательны.

```python
from typing import Required, TypedDict


class PatchUser(TypedDict, total=False):
    name: Required[str]
    email: str
    phone: str
```

`total=False` означает, что поля в целом необязательны, но `Required[...]` позволяет выделить те, без которых нельзя.

Это удобно для patch-структур, частичных обновлений, форм и конфигов.

## Аннотации атрибутов объектов

Типизация касается не только словарей, но и нормальных классов.

```python
class Product:
    id: int
    title: str
    price: float

    def __init__(self, id: int, title: str, price: float) -> None:
        self.id = id
        self.title = title
        self.price = price
```

Такие аннотации полезны для читаемости и для IDE. Особенно они хороши в классах, где поля появляются в `__init__`, в dataclass или в модели данных.

Если поле относится ко всему классу, а не к отдельному объекту, это уже тема `ClassVar`, но общая логика та же: мы явно документируем структуру.

## collections.abc и идея интерфейса коллекции

Иногда нас интересует не конкретно `list`, а более широкая идея: "последовательность", "итерируемое", "отображение".

Например, функция не обязана принимать именно список:

```python
from collections.abc import Iterable

def print_all(items: Iterable[str]) -> None:
    for item in items:
        print(item)
```

Теперь ей подойдет список, кортеж, множество, генератор и вообще любой итерируемый объект строк.

Это хороший инженерный сдвиг: типы помогают думать не только про данные, но и про интерфейс поведения.

Часто встречаются:

- `Iterable[T]`
- `Sequence[T]`
- `Mapping[K, V]`
- `Callable[...]`

Они обычно выражают намерение лучше, чем жесткая привязка к конкретной реализации.

---

## Главные правила

- Для коллекций почти всегда указывай тип элементов.
- Если словарь имеет фиксированную форму, лучше использовать `TypedDict`, а не размытый `dict[str, object]`.
- Для вложенных структур не бойся псевдонимов типов: они делают код короче и яснее.
- Если функция работает с интерфейсом, а не с конкретным контейнером, часто полезнее `Iterable` или `Sequence`, чем жесткий `list`.
- Аннотации атрибутов особенно важны в моделях данных и больших классах.
