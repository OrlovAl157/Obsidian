# 🧰 Модуль typing: Union, Optional, Any и другие базовые инструменты

> [!info] Коротко
> Модуль `typing` и современный синтаксис типов дают словарь, на котором строится почти вся практическая типизация в Python. Здесь живут не только `Union` и `Optional`, но и инструменты для более точного описания значений: `Any`, `NoReturn`, `TypeAlias`, `Literal`, `Final`, `ClassVar`.

## Справка по теме

| Инструмент | Пример | Когда нужен |
|---|---|---|
| `Union` | `int | str` | значение может быть одного из нескольких типов |
| `Optional` | `str | None` | значение может отсутствовать |
| `Any` | `data: Any` | тип временно неизвестен или слишком общий |
| `NoReturn` | `-> NoReturn` | функция не возвращается нормально |
| `TypeAlias` | `UserId: TypeAlias = int` | вынести сложный тип в имя |
| `Literal` | `Literal["GET", "POST"]` | ограничить конкретными значениями |
| `Final` | `PI: Final = 3.14` | показать, что значение не должно меняться |
| `ClassVar` | `role: ClassVar[str]` | атрибут уровня класса |

## Union и новый синтаксис через вертикальную черту

`Union` означает: значение может быть одним из нескольких типов.

Старый синтаксис:

```python
from typing import Union

def stringify(value: Union[int, float]) -> str:
    return str(value)
```

Современный синтаксис:

```python
def stringify(value: int | float) -> str:
    return str(value)
```

Оба варианта выражают одну и ту же идею. Новый короче и обычно читается лучше.

Использовать `Union` стоит там, где это отражает реальную модель данных. Например, когда функция правда принимает несколько допустимых форм входа:

```python
def load_user(user_id: int | str) -> str:
    return f"user={user_id}"
```

Но `Union` не должен превращаться в свалку всего подряд. Когда аргумент принимает пять разных типов, это уже повод спросить: а функция точно делает одну понятную вещь?

> [!tip] Хорошее правило
> `Union` удобен, если варианты действительно равноправны и понятны. Если тип становится длинным и мутным, часто лучше переработать API.

## Optional и None

`Optional[T]` — это сокращение для `T | None`. То есть значение либо есть, либо его нет.

```python
def find_name(user_id: int) -> str | None:
    if user_id == 1:
        return "Alice"
    return None
```

Старый вариант:

```python
from typing import Optional

def find_name(user_id: int) -> Optional[str]:
    ...
```

Ключевая идея не в синтаксисе, а в поведении: если у тебя `str | None`, то после вызова функции нельзя сразу бездумно делать `upper()`, `strip()` и другие строковые операции.

Нужно сузить тип:

```python
name = find_name(1)
if name is not None:
    print(name.upper())
```

Это как раз тот случай, где аннотация помогает поймать реальные ошибки раньше запуска.

## Any

`Any` — это "дырка" в системе типов. Он говорит: здесь проверка ослабляется.

```python
from typing import Any

def debug_print(data: Any) -> None:
    print(data)
```

`Any` полезен, когда:

- ты работаешь со старым кодом и типизация внедряется постепенно;
- приходят данные из внешнего мира, и тип еще не стабилизирован;
- библиотека слабо типизирована;
- ты сознательно оставляешь гибкость на границе системы.

Но злоупотреблять `Any` не стоит. Чем его больше, тем меньше смысла в типизации вообще.

Плохой стиль:

```python
def process(x: Any, y: Any) -> Any:
    return x + y
```

Так аннотации есть формально, но пользы почти ноль.

> [!warning] Важно
> `Any` не "универсальный хороший тип", а компромисс. Он нужен, но его стоит держать под контролем.

## NoReturn

`NoReturn` используют для функций, которые не завершаются обычным `return`. Например, всегда выбрасывают исключение или бесконечно работают.

```python
from typing import NoReturn

def fail(message: str) -> NoReturn:
    raise RuntimeError(message)
```

Это полезно не только как документация. Статический анализатор понимает, что после такого вызова код ниже может быть недостижим.

---

## TypeAlias

Когда тип длинный или часто повторяется, его стоит назвать.

```python
from typing import TypeAlias

JsonValue: TypeAlias = str | int | float | bool | None
JsonDict: TypeAlias = dict[str, JsonValue]
```

Преимущество тут не в "магии", а в читаемости. Сигнатура:

```python
def parse_config(data: JsonDict) -> None:
    ...
```

намного понятнее, чем огромный союз типов прямо в параметре.

## Literal

`Literal` используют, когда важно не просто семейство типов, а конкретные допустимые значения.

```python
from typing import Literal

def send(method: Literal["GET", "POST"]) -> None:
    ...
```

Здесь строка должна быть не любой, а строго одной из перечисленных.

Это особенно полезно для:

- режимов работы;
- имен команд;
- фиксированных флагов;
- небольших внутренних DSL.

## Final

`Final` показывает намерение: значение не должно переопределяться.

```python
from typing import Final

PI: Final = 3.14159
APP_NAME: Final[str] = "MyProject"
```

Python сам по себе не всегда жестко это запретит, но type checker поймет, что менять такое имя не надо.

Это хороший способ обозначать константы и защищать важные значения от случайной перезаписи.

## ClassVar

`ClassVar` используют для атрибутов, которые принадлежат классу, а не экземпляру.

```python
from typing import ClassVar

class User:
    role: ClassVar[str] = "member"

    def __init__(self, name: str) -> None:
        self.name = name
```

`role` здесь не индивидуальное поле объекта, а общее свойство класса. Это полезно для читаемости и особенно для dataclass-подобных структур, где важно разделять поля экземпляра и служебные атрибуты класса.

## Современный синтаксис и версии Python

В современном Python чаще пишут так:

```python
def get_status(flag: bool) -> str | None:
    ...
```

а не так:

```python
from typing import Optional

def get_status(flag: bool) -> Optional[str]:
    ...
```

То же самое касается коллекций:

- `list[str]` вместо `List[str]`
- `dict[str, int]` вместо `Dict[str, int]`

Но нужно помнить о версиях Python. Если ты работаешь со старой средой, новый синтаксис может быть недоступен или требовать строковых аннотаций.

> [!note] Практический ориентир
> Для личных заметок и современного кода разумно делать акцент на синтаксисе `|` и встроенных generic-типах. Старые формы полезно знать для чтения чужого кода.

## Главные правила

- `Union` нужен для реальных альтернатив, а не для скрытия неясной логики.
- `Optional[T]` и `T | None` означают одно и то же.
- `Any` используй осознанно и лучше на границах системы.
- `Literal` полезен, когда допустимы только конкретные значения, а не любой объект типа `str`.
- `Final` и `ClassVar` помогают не столько рантайму, сколько ясности модели.
