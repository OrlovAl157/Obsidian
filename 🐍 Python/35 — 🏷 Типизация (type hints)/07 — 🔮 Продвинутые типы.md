# 🔮 Продвинутые типы: ParamSpec, TypedDict, NewType

> Это продвинутые инструменты типизации для специальных случаев. **TypedDict** — для типизации словарей с известными ключами. **NewType** — для создания отличных друг от друга типов (например, UserId и ProductId оба int, но это разные вещи). **ParamSpec** — для сохранения сигнатуры функции при создании декораторов.

## 📖 TypedDict — типизированные словари

TypedDict позволяет типизировать словари с известными ключами и типами:

```python
from typing import TypedDict

class User(TypedDict):
    """Словарь пользователя"""
    name: str
    email: str
    age: int

# Использование
user: User = {
    "name": "Alice",
    "email": "alice@example.com",
    "age": 30
}

# mypy проверит ключи и типы!
print(user["name"])  # ✅ str
print(user["email"])  # ✅ str
print(user["phone"])  # ❌ Error: ключа нет в User
```

**Optional ключи:**

```python
from typing import TypedDict, Required, NotRequired

class User(TypedDict):
    name: str
    email: str
    phone: NotRequired[str]  # Опциональный ключ (Python 3.11+)

# Или старый стиль
class User(TypedDict, total=False):
    name: str
    email: str
    phone: str  # Все поля опциональны
```

## 🆕 NewType — новые типы

NewType создаёт новый тип на основе существующего:

```python
from typing import NewType

UserId = NewType('UserId', int)
ProductId = NewType('ProductId', int)

def get_user(user_id: UserId) -> str:
    return f"User {user_id}"

def get_product(product_id: ProductId) -> str:
    return f"Product {product_id}"

# ✅ Правильное использование
user = get_user(UserId(1))
product = get_product(ProductId(1))

# ❌ mypy скажет ошибка — передали неправильный тип!
user = get_user(ProductId(1))  # ❌ Error
```

**Важно:** NewType работает только в type checker! При выполнении это просто int:

```python
UserId = NewType('UserId', int)
uid: UserId = UserId(42)

print(uid)  # 42 (просто int)
print(type(uid))  # <class 'int'>
```

## 📝 ParamSpec — сигнатуры функций

ParamSpec сохраняет сигнатуру функции при создании декораторов:

```python
from typing import ParamSpec, TypeVar, Callable

P = ParamSpec('P')
R = TypeVar('R')

def log_calls(func: Callable[P, R]) -> Callable[P, R]:
    """Декоратор, сохраняющий сигнатуру"""
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
        print(f"Calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper  # type: ignore

@log_calls
def add(a: int, b: int) -> int:
    return a + b

@log_calls
def greet(name: str) -> str:
    return f"Hello, {name}!"

# mypy знает сигнатуру!
result = add(1, 2)  # ✅ mypy знает, что это int
message = greet("Alice")  # ✅ mypy знает, что это str
```

## 💻 Примеры

### Пример 1: TypedDict для API ответа

```python
from typing import TypedDict

class UserResponse(TypedDict):
    id: int
    name: str
    email: str
    is_active: bool

class ApiClient:
    def get_user(self, user_id: int) -> UserResponse:
        # В реальности это API вызов
        return {
            "id": user_id,
            "name": "Alice",
            "email": "alice@example.com",
            "is_active": True
        }

# Использование
client = ApiClient()
user = client.get_user(1)

print(user["name"])  # ✅ mypy знает, что это str
print(user["active"])  # ❌ Error: ключа нет
```

### Пример 2: NewType для ID’ов

```python
from typing import NewType

UserId = NewType('UserId', int)
OrderId = NewType('OrderId', int)

class User:
    def __init__(self, user_id: UserId, name: str):
        self.user_id = user_id
        self.name = name

class Order:
    def __init__(self, order_id: OrderId, user_id: UserId):
        self.order_id = order_id
        self.user_id = user_id

# ✅ Правильное использование
user = User(UserId(1), "Alice")
order = Order(OrderId(100), UserId(1))

# ❌ mypy скажет ошибка
# order = Order(UserId(1), OrderId(100))  # Перепутали!
```

### Пример 3: ParamSpec для универсального декоратора

```python
from typing import ParamSpec, TypeVar, Callable
import time

P = ParamSpec('P')
R = TypeVar('R')

def measure_time(func: Callable[P, R]) -> Callable[P, R]:
    """Декоратор, который измеряет время выполнения"""
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
        start = time.time()
        result = func(*args, **kwargs)
        end = time.time()
        print(f"{func.__name__} took {end - start:.2f}s")
        return result
    return wrapper  # type: ignore

@measure_time
def slow_operation(seconds: int) -> str:
    time.sleep(seconds)
    return "Done!"

@measure_time
def add_numbers(a: int, b: int) -> int:
    return a + b

# Использование
result = slow_operation(1)  # Сигнатура сохранена!
total = add_numbers(5, 3)  # Сигнатура сохранена!
```

### Пример 4: TypedDict с опциональными ключами

```python
from typing import TypedDict, NotRequired

class UserProfile(TypedDict):
    """Профиль пользователя с опциональными полями"""
    name: str
    email: str
    bio: NotRequired[str]
    avatar_url: NotRequired[str]

# Использование
profile1: UserProfile = {
    "name": "Alice",
    "email": "alice@example.com",
    "bio": "Python developer"
}

profile2: UserProfile = {
    "name": "Bob",
    "email": "bob@example.com"
    # bio отсутствует — OK, так как NotRequired
}

# ❌ Error: ключ name обязателен
# profile3: UserProfile = {"email": "test@example.com"}
```

## ⚠️ Частые ошибки

```python
from typing import TypedDict, NewType

# ❌ Мутирование TypedDict (нарушает типизацию)
class User(TypedDict):
    name: str
    email: str

user: User = {"name": "Alice", "email": "alice@example.com"}
user["phone"] = "123-456"  # ❌ Ошибка типов!

# ✅ Создать новый словарь
user_with_phone: dict[str, str] = {**user, "phone": "123-456"}

# ❌ Забыли скобки в NewType
UserId = NewType('UserId', int)
uid: UserId = 42  # ❌ Error! Нужно UserId(42)

# ✅ Правильно
uid: UserId = UserId(42)

# ❌ Думать, что NewType создаёт новый класс
UserId = NewType('UserId', int)
print(type(UserId(42)))  # <class 'int'>, не UserId!

# ✅ NewType работает только в type checker

# ❌ TypedDict без явного наследования (Python < 3.8)
User = TypedDict({'name': str, 'email': str})  # Старый стиль

# ✅ Современный стиль
class User(TypedDict):
    name: str
    email: str
```

## ✅ Главные правила

✅ **TypedDict для словарей** — когда знаешь ключи и типы  
✅ **NewType для отличия типов** — UserId vs ProductId  
✅ **ParamSpec для декораторов** — сохраняет сигнатуру функции  
✅ **TypedDict требует **всех ключей** — используй NotRequired для опциональных  
✅ **NewType только для type checker** — в runtime это базовый тип  
✅ **Не мутируй TypedDict** — создавай новые словари
