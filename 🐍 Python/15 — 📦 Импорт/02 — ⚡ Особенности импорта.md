## Справка по особенностям

| Явление | Описание | Проблема |
|---------|---------|----------|
| Анализ функции | Python анализирует функцию до её вызова | Ограничивает синтаксис внутри функции |
| `import *` в функции | Запрещено — компилятор не может знать, какие имена будут | `SyntaxError: import * only allowed at module level` |
| `__name__` | Имя текущего модуля; `__main__` если запущен напрямую | Используется для условного выполнения кода |
| `sys.modules` | Кэш всех загруженных модулей | Модуль загружается только один раз |
| Циклический импорт | Два модуля импортируют друг друга | Модуль может быть не полностью инициализирован |
| `__all__` | Список имён, экспортируемых `from module import *` | Без него экспортируются все публичные имена |
| `__file__` | Путь к файлу модуля | Полезен для работы с файлами относительно модуля |

## 🔧 Как Python анализирует функцию

Когда ты определяешь функцию, Python не выполняет её код сразу. Вместо этого он **анализирует** функцию, чтобы понять:

- Какие локальные переменные будут в функции?
- Какие имена будут доступны?
- Какие глобальные переменные используются?

Это происходит во время **компиляции**, до того как функция будет вызвана.

**Пример:**

```python
def my_func():
    x = 10
    y = 20
```

Когда Python видит эту функцию, он анализирует её и "запоминает": в этой функции будут локальные переменные `x` и `y`. Функция ещё не выполнена, но таблица локальных имён уже подготовлена.

**Зачем это нужно?** Потому что Python компилирует функцию в байт-код и должен заранее знать, где искать переменные: в локальной области или в глобальной.

```python
x = 100  # глобальная переменная

def func():
    x = 10  # локальная переменная
    print(x)  # какой x использовать? локальный или глобальный?
```

Python должен знать это заранее, при компиляции функции, а не при её выполнении.

## ❌ Почему `from module import *` в функции → SyntaxError

**Проблема:** когда ты пишешь `from module import *`, Python должен заранее (при анализе функции) знать, какие имена будут добавлены в локальную область.

Но со звёздочкой это **невозможно** — список имён определяется только во время выполнения!

**Пример:**

```python
def my_func():
    from math import *
    # Какие имена здесь будут? pi? sqrt? sin? cos?
    # Неизвестно до выполнения!
```

**Процесс:**

1. **При компиляции функции** Python видит `from math import *`
2. Думает: "Нужно добавить какие-то имена"
3. Но какие именно? Это зависит от того, что в модуле `math`, и это определяется только при выполнении
4. Python не может подготовить таблицу локальных имён
5. **Синтаксическая ошибка:** `SyntaxError: import * only allowed at module level`

**Сравнение:**

```python
def func1():
    from math import sqrt, sin  # OK! Python знает, что будут sqrt и sin
    print(sqrt(4))

def func2():
    from math import *  # ОШИБКА! Python не знает, какие будут имена
    print(sqrt(4))
```

**Правильные варианты:**

```python
# Вариант 1: импортировать конкретные имена
def func():
    from math import sqrt, sin
    print(sqrt(4))

# Вариант 2: импортировать модуль
def func():
    import math
    print(math.sqrt(4))

# Вариант 3: импортировать на уровне модуля, до функции
from math import sqrt, sin

def func():
    print(sqrt(4))
```

**Практический совет:** никогда не используй `from module import *`, но уж тем более не используй в функциях!

## 🎯 Точка входа `if __name__ == '__main__'`

**Точка входа** — это паттерн, который позволяет отличить запуск модуля напрямую от его импорта.

### Переменная `__name__`

Каждый модуль имеет встроенную переменную `__name__`, которая содержит имя модуля:

- **Если модуль запущен напрямую:** `__name__ = '__main__'`
- **Если модуль импортирован из другого модуля:** `__name__ = '<имя модуля>'`

**Пример:**

```python
# utils.py

def greet():
    print("Hello!")

print(f"Значение __name__: {__name__}")
```

**При запуске напрямую:**

```bash
python utils.py
```

Выведет:

```
Значение __name__: __main__
```

**При импорте в другой модуль:**

```python
# main.py

import utils  # выведет: Значение __name__: utils
```

### Паттерн `if __name__ == '__main__'`

Этот паттерн позволяет запускать код **только если модуль запущен напрямую**, но не запускать его при импорте:

```python
# utils.py

def greet(name):
    print(f"Hello, {name}!")

# Этот код запустится только если utils.py запущен напрямую
if __name__ == '__main__':
    greet("World")
    greet("Python")
```

**Запуск напрямую:**

```bash
python utils.py
```

Выведет:

```
Hello, World!
Hello, Python!
```

**При импорте:**

```python
# main.py

from utils import greet

# Сюда попадёт greet, но не выполнятся вызовы внутри if __name__ == '__main__'
greet("John")
```

Выведет:

```
Hello, John!
```

(Вывод из `utils.py` не появится)

### Почему это важно

Без этого паттерна:

```python
# utils.py

def process_data(data):
    return sorted(data)

# Опасно: выполняется при импорте!
result = process_data([3, 1, 2])
print(f"Результат: {result}")
```

```python
# main.py

import utils  # выведет: Результат: [1, 2, 3]
# Код из utils выполнился, хотя мы только импортировали функцию!
```

**Правильно:**

```python
# utils.py

def process_data(data):
    return sorted(data)

# Безопасно: выполняется только если запущен напрямую
if __name__ == '__main__':
    result = process_data([3, 1, 2])
    print(f"Результат: {result}")
```

```python
# main.py

import utils  # ничего не выведет
utils.process_data([5, 4, 3])  # мы сами вызываем функцию
```

### Лучшая практика

Всегда оборачивай вызовы функций в `if __name__ == '__main__'`:

```python
def main():
    # Основная логика программы
    data = [3, 1, 4, 1, 5, 9, 2, 6]
    result = process_data(data)
    print(f"Результат: {result}")

if __name__ == '__main__':
    main()
```

Это позволяет использовать модуль как библиотеку (импортировать функции) и одновременно запускать его как самостоятельный скрипт.

## 💾 Кэширование модулей в sys.modules

Когда Python загружает модуль, он сохраняет его в словаре `sys.modules`:

```python
import sys

print('math' in sys.modules)  # True (если импортировали math)
```

**Зачем это нужно?** Чтобы не загружать модуль повторно при каждом импорте.

**Пример:**

```python
# utils.py

print("Загрузка модуля utils")

def greet():
    print("Hello!")
```

```python
# main.py

import utils        # выведет: Загрузка модуля utils
import utils        # ничего не выведет (модуль уже в кэше)
from utils import * # ничего не выведет
```

При первом импорте код модуля выполняется, при повторных — модуль берётся из кэша `sys.modules` и код не выполняется.

**Проверить кэш:**

```python
import sys
import math

print('math' in sys.modules)  # True
print(sys.modules['math'])    # <module 'math' ...>
```

**Очистить кэш (опасно!):**

```python
import sys
del sys.modules['utils']
# Следующий импорт загрузит модуль заново
```

Обычно кэш трогают только в тестировании для перезагрузки модулей.

## 🔄 Циклические импорты

**Циклический импорт** — когда два модуля импортируют друг друга.

**Пример:**

```python
# a.py

import b

print("Модуль a загружен")
```

```python
# b.py

import a

print("Модуль b загружен")
```

```python
# main.py

import a
```

Что произойдёт?

1. `import a` загружает модуль `a`
2. `a` содержит `import b`, поэтому загружается модуль `b`
3. `b` содержит `import a`, но `a` уже в процессе загрузки!
4. Python берёт частично загруженный модуль `a` из кэша
5. `a` может быть ещё не полностью инициализирован — могут быть ошибки

**Ошибка при циклическом импорте:**

```python
# module_a.py

from module_b import func_b

def func_a():
    return func_b() + 1
```

```python
# module_b.py

from module_a import func_a

def func_b():
    return 1
```

```python
# main.py

import module_a  # ошибка! AttributeError или ImportError
```

**Как избежать циклических импортов:**

**Способ 1: импортировать локально**

```python
# module_b.py

def func_b():
    from module_a import func_a  # импортируем только когда нужно
    return func_a() + 1
```

**Способ 2: создать третий модуль**

```python
# common.py
def shared_func():
    return 42

# module_a.py
from common import shared_func

# module_b.py
from common import shared_func
```

**Способ 3: использовать `if __name__ == '__main__'`**

```python
# module_a.py

def func_a():
    return 1

if __name__ == '__main__':
    from module_b import func_b
    print(func_b())
```

## 🏷️ Переменная __all__

**Переменная `__all__`** — это список, который указывает, какие имена экспортирует модуль при использовании `from module import *`.

**Пример без `__all__`:**

```python
# math_utils.py

def add(a, b):
    return a + b

def multiply(a, b):
    return a * b

_internal_helper = "не для общего использования"
```

```python
# main.py

from math_utils import *

print(add(2, 3))           # работает
print(multiply(2, 3))      # работает
print(_internal_helper)    # работает, но не должна была!
```

При `from ... import *` экспортируются все имена, которые не начинаются с подчёркивания.

**С использованием `__all__`:**

```python
# math_utils.py

__all__ = ['add', 'multiply']

def add(a, b):
    return a + b

def multiply(a, b):
    return a * b

def _internal_helper():
    return "не для общего использования"
```

```python
# main.py

from math_utils import *

print(add(2, 3))           # работает
print(multiply(2, 3))      # работает
print(_internal_helper)    # ошибка! NameError
```

**Лучшая практика:** определяй `__all__` в своих модулях, чтобы ясно указать публичный API:

```python
# my_module.py

__all__ = ['public_function', 'PublicClass']

def public_function():
    pass

class PublicClass:
    pass

def _private_function():
    pass
```

## 👤 Атрибуты модулей __name__ и __file__

Каждый модуль имеет встроенные атрибуты:

### `__name__`

Имя модуля:

```python
import math

print(math.__name__)  # 'math'
```

Для главного модуля:

```python
# main.py
print(__name__)  # '__main__' (если запущен напрямую)
```

### `__file__`

Путь к файлу модуля:

```python
import math

print(math.__file__)  # '/usr/lib/python3.x/lib-dynload/math.cpython-3x.so'
```

Для модуля в текущем проекте:

```python
# utils.py
print(__file__)  # 'utils.py' или полный путь
```

**Практический пример:**

```python
# config.py

import os
from pathlib import Path

# Получить папку, где лежит этот модуль
MODULE_DIR = Path(__file__).parent

# Получить путь к файлу конфигурации в той же папке
CONFIG_FILE = MODULE_DIR / 'settings.json'

with open(CONFIG_FILE) as f:
    settings = json.load(f)
```

Это полезно, чтобы работать с файлами относительно модуля, а не относительно текущей директории запуска.

## 💡 Лучшие практики

### ✅ Делай:

1. **Используй абсолютные импорты** (или явные относительные):
   ```python
   from shop.utils import format_price  # хорошо
   from .utils import format_price      # хорошо
   ```

2. **Импортируй конкретные имена**, не `import *`:
   ```python
   from math import sqrt, sin  # хорошо
   ```

3. **Оборачивай код в `if __name__ == '__main__'`**:
   ```python
   if __name__ == '__main__':
       main()
   ```

4. **Определяй `__all__` в модулях**:
   ```python
   __all__ = ['public_func', 'PublicClass']
   ```

5. **Используй относительные импорты внутри пакетов**:
   ```python
   from . import sibling
   from ..parent import something
   ```

### ❌ Не делай:

1. **`from module import *`**:
   ```python
   from math import *  # плохо!
   ```

2. **Циклические импорты**:
   ```python
   # a.py: from b import func
   # b.py: from a import func  # плохо!
   ```

3. **Импорты на уровне функции без необходимости**:
   ```python
   def func():
       import random  # если используется часто, импортируй в начале
   ```

4. **Давай модулям имена встроенных модулей**:
   ```python
   # math.py  # плохо!
   # json.py  # плохо!
   ```

5. **Обращайся к глобальным переменным модуля как `module.var = value`** (лучше менять через функции).

## 📚 Итоговая таблица

| Ситуация | Что делать |
|----------|-----------|
| Загрузить модуль и использовать его объекты | `import module` → `module.func()` |
| Часто используешь один-два объекта | `from module import func1, func2` → `func1()` |
| Нужно выполнить код только при запуске скрипта | `if __name__ == '__main__': main()` |
| Контролировать, что экспортирует модуль | Определи `__all__ = [...]` |
| Работать с файлами относительно модуля | Используй `__file__` и `Path` |
| Импортировать внутри пакета | Используй относительные импорты: `from . import` |
| Избежать циклических импортов | Импортируй локально в функциях или создай третий модуль |
