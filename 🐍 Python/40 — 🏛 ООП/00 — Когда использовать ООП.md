# 🏛 Когда использовать ООП

## 📌 Коротко

ООП оправдано когда нужно **объединить данные и операции над ними в единую абстракцию с контролем состояния и поведения**. Если этого нет — достаточно функций.

-----

## ✅ Используй ООП когда

- Нужно **инкапсулировать состояние** — данные и операции над ними логически связаны
- Нужно гарантировать **инвариант** — объект всегда в корректном состоянии
- Есть **жизненный цикл** объекта — создаётся, изменяется, уничтожается
- Нужно создавать **множество похожих объектов**
- Нужна **изоляция логики** и контроль над данными

```python
# ✅ Хороший кандидат для класса
class BankAccount:
    def __init__(self, balance):
        self.balance = balance  # состояние

    def withdraw(self, amount):
        if amount > self.balance:          # → контроль инварианта
            raise ValueError("Недостаточно средств")
        self.balance -= amount

    def deposit(self, amount):
        self.balance += amount
```

> 💡 **Инвариант** — это гарантия что объект всегда находится в корректном состоянии. Например баланс счёта никогда не уходит в минус. Это один из главных архитектурных аргументов в пользу класса.

-----

## ❌ Используй функции когда

- Простая задача — одно действие без побочных эффектов
- Данные и логика **не связаны** между собой
- Состояние не нужно инкапсулировать

```python
# ✅ Достаточно функции — состояние есть, но инкапсуляция не нужна
settings = {"theme": "dark"}

def change_theme(settings, theme):
    settings["theme"] = theme
```

> ⚠️ Наличие состояния само по себе не означает что нужен класс. Важно — нужно ли **логически объединить** данные и операции над ними.

-----

## 🔑 Ключевые вопросы перед выбором

```
1. Нужно ли объединить данные и операции? → класс
2. Нужно гарантировать корректность состояния (инвариант)? → класс
3. Будет много похожих объектов? → класс
4. Есть жизненный цикл объекта? → класс
5. Одно простое действие без инкапсуляции? → функция
```

-----

## 📊 Сравнение

|Критерий                  |Функции|ООП    |
|--------------------------|-------|-------|
|Инкапсуляция состояния    |❌      |✅      |
|Контроль инварианта       |❌      |✅      |
|Жизненный цикл объекта    |❌      |✅      |
|Множество похожих объектов|❌      |✅      |
|Простота                  |✅      |сложнее|


> ⚠️ Наследование — не главный аргумент в пользу ООП. В реальной архитектуре **композиция** часто лучше наследования.

## 🧠 Граница архитектурного мышления

```python
# Функция — достаточно если просто добавляем
def add_user(users, name):
    users.append(name)

# Класс — оправдан когда появляется логика и инвариант
class UserManager:
    def __init__(self):
        self._users = []

    def add_user(self, name):
        if name in self._users:       # → инвариант: нет дубликатов
            raise ValueError("Пользователь уже существует")
        self._users.append(name)

    def remove_user(self, name):
        self._users.remove(name)
```
