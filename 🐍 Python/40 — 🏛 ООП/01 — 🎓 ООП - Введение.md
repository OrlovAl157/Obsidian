## 📌 Коротко

> ООП — парадигма программирования, в которой программа строится из объектов, взаимодействующих друг с другом.

---

## 📖 Теория

### Что такое ООП?

**ООП (объектно-ориентированное программирование)** — парадигма программирования, в которой программа строится из объектов, взаимодействующих друг с другом.

Основное преимущество перед процедурным подходом — **изоляция кода на уровне классов**: код становится понятнее, проще в поддержке и повторном использовании.

---

### Класс и объект

|Понятие|Что это|Аналогия|
|---|---|---|
|**Класс**|Шаблон/описание|Чертёж дома|
|**Объект**|Конкретный экземпляр класса|Построенный дом по чертежу|

- **Класс** описывает структуру и поведение — сам по себе ничего не делает
- **Объект** — живой экземпляр класса с конкретными значениями атрибутов
- По одному классу можно создать сколько угодно объектов
- Каждый объект существует независимо: изменение одного не затрагивает другие

### 💻 Примеры кода

```python
class Car:
    def __init__(self, color, engine):
        self.color = color       # атрибут — состояние объекта
        self.engine = engine     # атрибут — состояние объекта

    def drive(self):             # метод — поведение объекта
        print(f"{self.color} автомобиль едет")

    def info(self):
        print(f"Цвет: {self.color}, двигатель: {self.engine}л")

# Создаём два независимых объекта
car1 = Car("чёрный", 2.9)
car2 = Car("жёлтый", 2.0)

car1.drive()   # → чёрный автомобиль едет
car2.drive()   # → жёлтый автомобиль едет
car1.info()    # → Цвет: чёрный, двигатель: 2.9л

# Изменение car1 не влияет на car2
car1.color = "красный"
car2.info()    # → Цвет: жёлтый, двигатель: 2.0л  (не изменился)
```

> 💡 Нельзя создать объект без класса — класс всегда первичен.

---

### ⚙️ Что такое self?

`self` — это ссылка на текущий объект. Python передаёт его автоматически при вызове метода.

```python
car1.drive()
# Фактически интерпретируется как:
Car.drive(car1)
```

Поэтому первый параметр любого метода — всегда `self`, даже если снаружи мы его не передаём вручную.

---

### ⚙️ Атрибут класса vs атрибут экземпляра

```python
class Dog:
    species = "Canis lupus"  # атрибут класса — общий для всех объектов

    def __init__(self, name):
        self.name = name     # атрибут экземпляра — у каждого свой

dog1 = Dog("Рекс")
dog2 = Dog("Бобик")

print(Dog.species)       # → Canis lupus  (обращение через класс)
print(dog1.species)      # → Canis lupus  (унаследовано от класса)
print(dog1.name)         # → Рекс
print(dog2.name)         # → Бобик

# Изменение атрибута класса затронет все объекты
Dog.species = "Собака"
print(dog1.species)      # → Собака
print(dog2.species)      # → Собака
```

||Атрибут класса|Атрибут экземпляра|
|---|---|---|
|Где задаётся|В теле класса|В `__init__` через `self`|
|Хранится|В классе|В каждом объекте отдельно|
|Общий?|Да, для всех объектов|Нет, у каждого свой|

## 4 принципа ООП

### 🔵 Абстракция

Выделяем только те атрибуты и методы, которые нужны для конкретной задачи. Скрываем всё лишнее — пользователь класса работает с простым интерфейсом, не зная деталей реализации.

**Аналогия:** водитель нажимает на газ и рулит — он не знает, как работает двигатель внутри. Интерфейс простой, детали скрыты.

```python
class Programmer:
    def __init__(self, name, language, experience):
        self.name = name
        self.language = language
        self.experience = experience
        # favorite_color, shoe_size и т.д. — не нужны для задачи

    def write_code(self):
        return f"{self.name} пишет на {self.language}"

    def is_senior(self):
        return self.experience >= 5


p = Programmer("Алиса", "Python", 6)
print(p.write_code())    # → Алиса пишет на Python
print(p.is_senior())     # → True
```

**Зачем нужна:**

- Упрощает работу с классом — не нужно знать детали реализации
- Позволяет менять реализацию, не трогая внешний код
- Снижает когнитивную нагрузку

---

### 🟡 Инкапсуляция

Объединяем данные и методы внутри класса. Скрываем внутреннее состояние от прямого доступа извне — данные меняются только через методы, которые контролируют корректность.

**Аналогия:** нельзя напрямую положить еду в желудок — только через рот. Прямой доступ опасен, поэтому он закрыт.

```python
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.__balance = balance    # __ → приватный атрибут

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("Сумма должна быть положительной")
        self.__balance += amount
        print(f"Пополнено на {amount}. Баланс: {self.__balance}")

    def withdraw(self, amount):
        if amount > self.__balance:
            raise ValueError("Недостаточно средств")
        self.__balance -= amount
        print(f"Снято {amount}. Баланс: {self.__balance}")

    def get_balance(self):          # контролируемый доступ
        return self.__balance


account = BankAccount("Иван", 1000)
account.deposit(500)                # → Пополнено на 500. Баланс: 1500
account.withdraw(200)               # → Снято 200. Баланс: 1300
print(account.get_balance())        # → 1300

# account.__balance = 1000000      # ❌ не сработает как ожидается
# print(account.__balance)         # ❌ AttributeError
```

**Уровни доступа в Python:**

|Синтаксис|Тип|Смысл|
|---|---|---|
|`self.name`|публичный|доступен откуда угодно|
|`self._name`|защищённый|соглашение "не трогай снаружи"|
|`self.__name`|приватный|name mangling, скрыт от внешнего кода|

**Зачем нужна:**

- Защищает данные от случайного изменения
- Позволяет добавлять валидацию при изменении атрибутов
- Скрывает детали реализации

### 🟢 Наследование

Создаём новый класс на основе существующего. Дочерний класс получает все атрибуты и методы родителя, а затем расширяет или переопределяет их.

**Аналогия:** класс `Стол` → дочерние классы `ОбеденныйСтол`, `ПисьменныйСтол`, `ЖурнальныйСтол`. Все они столы, но с разными особенностями.

```python
class Animal:                            # родительский класс
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def eat(self):
        print(f"{self.name} ест")

    def sleep(self):
        print(f"{self.name} спит")

    def describe(self):
        print(f"{self.name}, {self.age} лет")


class Dog(Animal):                       # наследуем от Animal
    def __init__(self, name, age, breed):
        super().__init__(name, age)      # вызываем __init__ родителя
        self.breed = breed               # добавляем свой атрибут

    def bark(self):                      # новый метод
        print(f"{self.name}: Гав!")

    def describe(self):                  # переопределяем метод родителя
        print(f"{self.name} ({self.breed}), {self.age} лет")


class Cat(Animal):
    def __init__(self, name, age, indoor):
        super().__init__(name, age)
        self.indoor = indoor

    def meow(self):
        print(f"{self.name}: Мяу!")


dog = Dog("Рекс", 3, "Овчарка")
dog.eat()        # → Рекс ест              (унаследовано)
dog.bark()       # → Рекс: Гав!            (своё)
dog.describe()   # → Рекс (Овчарка), 3 лет (переопределено)

cat = Cat("Мурка", 5, True)
cat.sleep()      # → Мурка спит            (унаследовано)
cat.meow()       # → Мурка: Мяу!           (своё)
```

**Проверка наследования:**

```python
print(isinstance(dog, Dog))     # → True
print(isinstance(dog, Animal))  # → True  (dog — это тоже Animal)
print(isinstance(cat, Dog))     # → False
```

**Зачем нужно:**

- Повторное использование кода — не пишем одно и то же дважды
- Иерархия классов отражает реальные связи между сущностями
- Легко расширять функциональность

> ⚠️ **Множественное наследование** (несколько родителей) возможно в Python, но может вызвать конфликты если у родителей есть методы с одинаковыми именами. Использовать осторожно.

### 🟠 Полиморфизм

Объекты разных классов могут иметь методы с одинаковым именем, но разной реализацией. Код работает с объектами через единый интерфейс, не зная их конкретного типа.

**Аналогия:** знак `+` для чисел — это сложение, для строк — конкатенация. Интерфейс один, поведение разное.

```python
class Dog:
    def speak(self):
        return "Гав!"

    def describe(self):
        return "Я собака"

class Cat:
    def speak(self):
        return "Мяу!"

    def describe(self):
        return "Я кошка"

class Duck:
    def speak(self):
        return "Кря!"

    def describe(self):
        return "Я утка"


# Полиморфизм в действии — один код работает с разными типами
animals = [Dog(), Cat(), Duck()]

for animal in animals:
    print(f"{animal.describe()} и говорю: {animal.speak()}")
# → Я собака и говорю: Гав!
# → Я кошка и говорю: Мяу!
# → Я утка и говорю: Кря!
```

**Полиморфизм через наследование:**

```python
class Shape:
    def area(self):
        raise NotImplementedError("Метод должен быть реализован в дочернем классе")

class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius

    def area(self):
        return 3.14 * self.radius ** 2

class Rectangle(Shape):
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def area(self):
        return self.width * self.height


shapes = [Circle(5), Rectangle(4, 6), Circle(3)]

for shape in shapes:
    print(f"Площадь: {shape.area()}")
# → Площадь: 78.5
# → Площадь: 24
# → Площадь: 28.26
```

**Зачем нужен:**

- Код становится гибким — легко добавлять новые типы, не меняя существующий код
- Один интерфейс для разных реализаций упрощает логику программы

## Быстрая шпаргалка

|Принцип|Суть|Ключевая идея|
|---|---|---|
|**Абстракция**|Показываем только нужное, скрываем лишнее|Упрощение интерфейса|
|**Инкапсуляция**|Данные и методы вместе, прямой доступ закрыт|Защита данных|
|**Наследование**|Дочерний класс расширяет родительский|Переиспользование кода|
|**Полиморфизм**|Один интерфейс — разные реализации|Гибкость и расширяемость|

---

## Когда использовать ООП?

**Используй ООП, когда:**

- Программа моделирует реальные сущности (пользователь, товар, заказ, банковский счёт)
- Нужно переиспользовать код через наследование
- Важна защита данных от случайных изменений
- Проект большой и будет развиваться со временем
- Несколько разработчиков работают над одним проектом

**Процедурный подход лучше, когда:**

- Простой скрипт или одноразовая утилита
- Нет сложных взаимосвязей между данными
- Важна максимальная простота и скорость написания кода
