# ✅ Статический анализатор mypy

> **mypy** — статический анализатор кода, автор которого — Гвидо ван Россум (создатель Python). mypy проверяет типы в коде **без его запуска**, находя несоответствия между аннотациями и фактическим использованием. Это мощный инструмент, но в этом файле мы сосредоточимся именно на проверке аннотаций типов.

## 📍 Первая проверка

Файл `main.py`:

```python
def add(a: int, b: int) -> int:
    return a + b

print(add(1, 2))
```

Запускаем:

```bash
mypy main.py
```

Результат:

```
Success: no issues found in 1 source file
```

mypy не нашёл ошибок — функция ожидает два `int` и возвращает `int`, код отработает как ожидается.

## 🔴 Ошибка в аргументах

Оператор `+` можно использовать не только для чисел, но и для конкатенации строк. Такое поведение не ожидается для функции `add()`. Если она получит строку вместо числа, код, использующий эту функцию, может завершиться с ошибкой:

```python
def add(a: int, b: int) -> int:
    return a + b

print(add('1', '2') / 2)
```

Приводит к исключению **при выполнении**:

```
TypeError: unsupported operand type(s) for /: 'str' and 'int'
```

mypy предотвращает такие ошибки **до запуска кода**:

```bash
main.py:4: error: Argument 1 to "add" has incompatible type "str"; expected "int"  [arg-type]
main.py:4: error: Argument 2 to "add" has incompatible type "str"; expected "int"  [arg-type]
Found 2 errors in 1 file (checked 1 source file)
```

## 🎭 Явная передача vs динамическое получение типов

Это **ключевой нюанс** работы mypy, который часто вызывает путаницу.

### Случай 1: Проверка isinstance() внутри функции НЕ спасает от ошибки mypy

Добавим в `add()` проверку аргументов через `isinstance`:

```python
def add(a: int, b: int) -> int:
    if not isinstance(a, int) or not isinstance(b, int):
        raise TypeError
    return a + b

print(add('1', '2') / 2)
```

**Несмотря на runtime-проверку**, mypy всё равно указывает на ошибку:

```
main.py:6: error: Argument 1 to "add" has incompatible type "str"; expected "int"  [arg-type]
main.py:6: error: Argument 2 to "add" has incompatible type "str"; expected "int"  [arg-type]
Found 2 errors in 1 file (checked 1 source file)
```

**Почему:** мы передаём строковые аргументы **явно**, несмотря на то что в аннотации указан `int`. mypy замечает это и предупреждает, что функция вызывается неправильно — `isinstance` проверка в теле функции не отменяет статическую проверку вызова.

### Случай 2: Динамическое получение типов НЕ вызывает ошибку mypy

```python
def add(a: int, b: int) -> int:
    if not isinstance(a, int) or not isinstance(b, int):
        raise TypeError
    return a + b

def to_str(a, b):
    return str(a), str(b)

a, b = to_str(1, 2)
print(add(a, b) / 2)
```

Здесь **никаких ошибок** со стороны mypy не будет:

```
Success: no issues found in 1 source file
```

**Почему так происходит:** мы не задаём аргументы явно, нарушая контракт функции, а получаем их из внешнего кода (`to_str`), где заведомо **неизвестно**, аргументы каких типов данных она вернёт (функция `to_str` без аннотаций — её возвращаемый тип неявно `Any`). Если это будет два целых числа — функция отработает как ожидается, если что-то другое — произойдёт ошибка **при выполнении**, но mypy не может это предсказать статически.

> **Вывод:** mypy ловит ошибки там, где типы **известны статически** и явно противоречат аннотации. Там, где типы приходят динамически (например, из функции без аннотаций → `Any`), mypy “доверяет” коду.

## 🔄 Несовпадение return type

Заменим аннотацию возвращаемого значения с `int` на `str`:

```python
def add(a: int, b: int) -> str:
    return a + b
```

mypy покажет:

```
main.py:2: error: Incompatible return value type (got "int", expected "str")  [return-value]
Found 1 error in 1 file (checked 1 source file)
```

Ошибка говорит: ожидается объект типа `str`, но поскольку выполняется сложение `a` и `b` типа `int`, фактически возвращаемое значение — `int`, что противоречит аннотации.

-----

## 📝 Обязательность аннотаций

Уберём у функции `add()` все аннотации:

```python
def add(a, b):
    return a + b
```

**По умолчанию mypy не требует обязательного наличия аннотаций:**

```
Success: no issues found in 1 source file
```

Если нужно, чтобы отсутствующие аннотации **тоже считались ошибкой**, используем опцию `--disallow-untyped-defs`:

```bash
mypy main.py --disallow-untyped-defs
```

Результат:

```
main.py:1: error: Function is missing a type annotation  [no-untyped-def]
Found 1 error in 1 file (checked 1 source file)
```

## 💻 Примеры

### Пример 1: Полная проверка директории

```bash
mypy .
```

Проверяет **все** Python-файлы в текущей директории и подпапках (не нужно перечислять каждый файл вручную — путь до файла, а не просто имя; если файл в той же директории, где вы находитесь, можно указать просто имя файла).

### Пример 2: Type narrowing с isinstance

```python
from typing import Union

def process(value: Union[int, str]) -> str:
    if isinstance(value, str):
        return value.upper()  # ✅ mypy знает, что это str
    else:
        return str(value)     # ✅ Это int, преобразуем
```

Здесь `isinstance` работает **вместе** с mypy, а не против него — потому что `value` **уже объявлен** как `Union[int, str]`, а не передаётся с явно неправильным типом.

### Пример 3: Разница между явной передачей и динамикой

```python
def strict_func(x: int) -> int:
    return x * 2

# ❌ mypy поймает — явная передача неправильного типа
strict_func("5")

# ✅ mypy НЕ поймает — тип приходит динамически (Any)
def get_value():  # без аннотаций!
    return "5"

value = get_value()
strict_func(value)  # mypy не ругается, хотя в runtime будет ошибка!
```

**Вывод:** отсутствие аннотаций у `get_value()` — это “дыра” в системе типов. Аннотируйте **все** функции для максимальной пользы от mypy.

### Пример 4: Опции командной строки

```bash
# Базовая проверка
mypy main.py

# Строгий режим — требовать аннотации везде
mypy main.py --disallow-untyped-defs

# Игнорировать отсутствующие импорты (для сторонних библиотек без stubs)
mypy main.py --ignore-missing-imports

# Максимальная строгость (включает много опций сразу)
mypy main.py --strict
```

## ⚠️ Частые ошибки

```python
# ❌ Думать, что isinstance() внутри функции защищает от ошибок mypy
def add(a: int, b: int) -> int:
    if not isinstance(a, int):
        raise TypeError
    return a + b

add('5', 3)  # ❌ mypy ВСЁ РАВНО укажет на ошибку!

# ✅ isinstance полезен для runtime, но не отменяет статическую проверку вызова

# ❌ Думать, что mypy проверяет типы во время выполнения
def add(a: int, b: int) -> int:
    return a + b

add("hello", "world")  # ✅ Python выполнит это нормально!
                       # ❌ Но mypy (при статическом анализе) укажет ошибку

# ❌ Забывать, что функции без аннотаций = дыра в системе типов
def helper(x):  # Нет аннотаций!
    return x * 2

result = helper("5")  # mypy не поймает проблему!

# ✅ Аннотируйте вспомогательные функции тоже
def helper(x: int) -> int:
    return x * 2

# ❌ Не запускать mypy вообще, полагаясь только на аннотации
# Аннотации без проверки — это просто комментарии, они не защищают код!

# ✅ Всегда запускайте mypy (в CI/CD, pre-commit hook)
```

## ✅ Главные правила

✅ **mypy проверяет типы статически** — до запуска программы  
✅ **`isinstance()` в теле функции НЕ отменяет** ошибку mypy при явно неправильном вызове  
✅ **Динамически полученные типы** (из функций без аннотаций) mypy не может проверить — это “слепая зона”  
✅ **Аннотируйте ВСЕ функции**, включая вспомогательные — иначе получите `Any` и потеряете защиту  
✅ **По умолчанию аннотации не обязательны** — используйте `--disallow-untyped-defs` для строгости  
✅ **`mypy .`** — проверка всей директории одной командой  
✅ **mypy не запускает код** — только анализирует, поэтому безопасен для CI/CD  
✅ **Существуют и другие анализаторы** — например, pyright и pylint
